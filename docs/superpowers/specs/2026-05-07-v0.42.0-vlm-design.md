# v0.42.0 Vision-Language Models (VLMs): Design

**Status:** Approved (pending user review of this spec)
**Target version:** v0.42.0
**Task:** #97 (originally "Modality: image/description"; reframed below)
**Date:** 2026-05-07

## Goal

Add support for vision-language models (VLMs) to muse via the existing `/v1/chat/completions` endpoint. Users send OpenAI-shape multimodal `messages` (text + `image_url` parts) and get text responses back. Bundled default is `smolvlm-256m-instruct` (~500MB, CPU-runnable); curated additions cover SmolVLM-Instruct, Qwen2-VL-2B, LLaVA-1.5-7B, and Qwen2-VL-7B.

## Decisions (locked)

| # | Question | Choice |
|---|---|---|
| Q1 | Endpoint shape | Extend existing `/v1/chat/completions`. No new URL. |
| Q2 | Runtime backend | `transformers.AutoModelForImageTextToText` (broad VLM family coverage). llama-cpp LLaVA deferred. |
| Q3 | MIME tag organization | Under existing `chat/completion` MIME with new `supports_vision` capability. NO new modality directory. |
| Q4 | Bundled default | `smolvlm-256m-instruct` (Apache 2.0, ~500MB, CPU-runnable). + 4 curated entries. |
| Q5 | Multi-image per message | Supported in v1 with `supports_multi_image` capability flag. |

**Reframing:** Task #97's original name "Modality: image/description (VLMs)" implied a new modality directory. The decision under Q3 is that VLMs are a *capability addition* to `chat/completion`, not a new modality. The task remains the unit of work but its title is misleading.

**Adopted defaults (no question asked, sensible defaults from existing precedent):**

- Image input format: `data:image/{png,jpeg,webp};base64,...` and `http(s)://...` via the existing `decode_image_input` async helper (with SSRF guard, MUSE_IMAGE_INPUT_MAX_BYTES cap).
- Capability mismatch: 400 error (mirrors image/generation `supports_img2img/inpainting/variations` precedent; differs from `supports_tools` which warn-and-passes).
- Streaming: works "for free" on the existing SSE path. Decode + validate runs synchronously before stream open.
- Token accounting: whatever `AutoProcessor` + model returns for prompt/completion token counts. No special image-token math.

## Architecture & data flow

```
Client (OpenAI SDK)
    POST /v1/chat/completions
    body.messages = [{
        role: user,
        content: [
            {type: text, text: "What's in this?"},
            {type: image_url, image_url: {url: "data:..."}}
        ]
    }]
        |
        v
chat_completion/routes.py: ChatCompletionRequest
  (existing model; messages: list[dict] passthrough, no schema change)
        |
        v
NEW pre-dispatch step (chat_completion/routes.py)
  1. Walk messages, collect content parts of type=="image_url"
  2. If any parts found:
       a. Check model.supports_vision (else 400 vision_not_supported)
       b. Check max images-per-message vs supports_multi_image
          (else 400 too_many_images)
       c. Validate each part has image_url.url field
          (else 400 invalid_content_part)
       d. Reject unknown content types (video_url, audio_url, ...)
          (else 400 unsupported_content_type)
       e. Decode each url via decode_image_input (async)
          (else 400 invalid_image)
       f. Inline-rewrite the part: {type: image_url, image_url: {url}}
          becomes {type: image, image: <PIL.Image>}
          (muse-internal shape; backends consume by type tag)
        |
        v
ChatModel.chat() / chat_stream()  (existing protocol; messages: list[dict] passthrough)
        |
        v
NEW runtime: HFVisionLanguageModel
  (chat_completion/runtimes/transformers_vlm.py)
  - lazy-imports torch, transformers, Pillow (deferred-imports pattern)
  - loads AutoModelForImageTextToText + AutoProcessor
  - chat(messages):
      walks message content: text parts → joined text;
      image parts (already PIL.Image after route layer) → images list
      processor(text=..., images=...) → inputs
      model.generate(**inputs, max_new_tokens=...) → output_ids
      processor.batch_decode(output_ids) → text
      pack into ChatResult (existing dataclass)
  - chat_stream(messages):
      TextIteratorStreamer in producer thread;
      consumer yields ChatChunk (existing dataclass)
        |
        v
chat_completion/codec.py: result_to_openai_dict / chunk_to_sse_data
  (UNCHANGED)
        |
        v
SSE or JSON response
```

### Why this shape

The new code is concentrated in two places: the route's pre-dispatch image-decode/validate step, and the new VLM runtime. The protocol, codec, client, and registry are untouched. Text-only chat requests are byte-identical to v0.41.x (verified by the existing test suite as a regression watchdog).

The "decode at the route layer, not in the runtime" choice means: image fetch errors surface as 400s with clean OpenAI-shape envelopes (not 500s from inside a backend), the runtime stays sync-friendly without its own httpx/SSRF logic, and future runtimes (e.g. llama-cpp LLaVA in v.next) reuse the same already-decoded inputs.

## Capability schema

New manifest fields under `capabilities`:

| Field | Type | Default | Effect |
|---|---|---|---|
| `supports_vision` | bool | absent/false | When false: route returns 400 `vision_not_supported` if any `image_url` parts are present in the request |
| `supports_multi_image` | bool | absent/false | When false: route returns 400 `too_many_images` if any single message has >1 `image_url` parts |

These join the existing zoo (`supports_tools`, `supports_img2img`, `supports_inpainting`, `supports_variations`, etc.). Surfaced verbatim in `/v1/models` so SDK clients can branch on them client-side.

## Error handling

### Pre-dispatch error matrix

| Condition | Status | Error code | Message |
|---|---|---|---|
| `image_url` part present, model `supports_vision != True` | 400 | `vision_not_supported` | `model {id} does not support vision input; pick a model with supports_vision=true` |
| Any single message has >1 `image_url` part, model `supports_multi_image != True` | 400 | `too_many_images` | `model {id} accepts only 1 image per message; got N` |
| `content` part has `type: image_url` but no `image_url.url` field | 400 | `invalid_content_part` | `image_url part missing required url field` |
| `content` part `type` is not in `{text, image_url}` | 400 | `unsupported_content_type` | `content type {t!r} not supported; allowed: text, image_url` |
| `decode_image_input` raises `ValueError` | 400 | `invalid_image` | `could not decode image: {ve}` |

All errors use the OpenAI-shape envelope (`{"error": {"code", "message", "type"}}`).

### Streaming discipline

All five errors surface synchronously **before** `EventSourceResponse` is constructed. Once a stream starts, OpenAI clients don't expect `event: error` and will hang or misparse. Decode + validate runs entirely before stream open; once the stream starts, any backend error (model.generate fails) follows the existing chat_completion error-chunk path.

### Edge cases that just work

- Pure-text request to a VLM model: no image_url parts, no decode step, model handles text-only natively.
- Text-only request to text-only chat model: byte-identical to v0.41.x.
- System message with `image_url`: passed through; model decides.
- `content: "string"` (legacy OpenAI shape): no decode step, normal text flow, backward-compatible.
- `http(s)://` image fetch: existing SSRF guard refuses private/loopback/link-local IPs unless `MUSE_ALLOW_PRIVATE_FETCH=1`.
- Large images: `MUSE_IMAGE_INPUT_MAX_BYTES` (default 10MB) applies per-image.

### Capability mismatch logging precedent

| Capability | Mismatch behavior |
|---|---|
| `supports_tools: False` | warn + pass (model might still produce sensible text without structured tool_calls) |
| `supports_img2img / inpainting / variations / vision` | 400 reject (model literally can't process the input) |
| `supports_multi_image: False` (with >1 image) | 400 reject |

## Release plan

### Version bump rationale

`0.41.2 -> 0.42.0` (minor bump). New public capability flags + new generic runtime + meaningful new behavior on `/v1/chat/completions`. Patch (`0.41.3`) would understate the surface; major is reserved for stability commitments not yet made.

### Release ritual

1. Bump `pyproject.toml`: `version = "0.42.0"`.
2. Update CLAUDE.md (chat/completion bullet + new paragraph documenting capability flags).
3. Update README.md (vision example block).
4. Run fast lane + slow lane locally; expect ~2992 passed.
5. Live-verify: `muse pull smolvlm-256m-instruct && muse models probe smolvlm-256m-instruct` succeeds; `curl /v1/chat/completions` with a small data-URL image returns valid response.
6. Commit: `feat(chat/completion): vision-language model support (VLMs) v0.42.0`.
7. Tag: `git tag -a v0.42.0 -m "v0.42.0: vision-language models"`.
8. Push: `git push origin main && git push origin v0.42.0`.
9. GitHub release: `gh release create v0.42.0` with notes.

### Migration / breaking changes: none

- Text-only chat requests are byte-identical to v0.41.x.
- Existing GGUF models keep working unchanged (manifests don't gain `supports_vision: true`, route never tries to decode images).
- Catalog format: backward-compatible. New capability flags are additive.
- Wire format: `/v1/chat/completions` accepts MORE inputs (multimodal content arrays) but rejects the same set of malformed inputs as before.

## Out of scope for v0.42.0

Explicitly deferred:

- llama-cpp-python LLaVA chat handlers. transformers covers SmolVLM + Qwen2-VL which are the bigger gap.
- VLM tool-use (`supports_vision: true` AND `supports_tools: true`). Combinatoric edge cases; defer until a real use case.
- Video parts (`type: video_url`). Different problem domain.
- Audio parts in chat (`type: input_audio`). OpenAI realtime API shape, deferred.
- Image-side response (`type: image_url` in assistant `content`). VLMs in v0.42.0 are unimodal-output (text only).
- Cross-modality consolidation #236 (still dormant per prior session).

## Open questions / future work

- Token accounting refinement: OpenAI charges per-image at a fixed rate (85 + tile_count*N tokens). Our v0.42.0 accounting reports whatever the processor + model returns, which may differ from OpenAI's math. Future polish: surface a `usage.prompt_tokens.image_tokens` breakdown when the processor exposes it.
- VLM benchmark suite: no benchmarks shipping in v0.42.0. A future `tests/integration/test_vlm_quality.py` could pin minimum-quality assertions on a small public benchmark (e.g. POPE, MM-Vet) so regressions surface.
- Curated entry quality drift: SmolVLM-Instruct's metric_gb estimate (5GB) is conservative; real fp16 footprint may be 4.0-4.5GB. Self-healing measurement on first probe (existing v0.40.0+ infra) handles this transparently.
- Idefics2 / SmolVLM2 family upgrade path: as new VLM architectures land, the HF resolver's sniff list needs occasional extension. Treat as routine maintenance.
