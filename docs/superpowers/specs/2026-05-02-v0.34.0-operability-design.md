# v0.34.0 Operability Cleanup Design

**Date:** 2026-05-02
**Status:** Approved for plan
**Source:** Important findings from the v0.32.0 code review (8 of 9; finding #10 was implicitly resolved when `decode_image_input` became async in v0.33.0).

## Goal

Address eight Important findings from the post-v0.32.0 review. The bugs are all in admin and supervisor concurrency, async resource lifecycles, and operator ergonomics. None are user-visible API changes; all are visible only under load or after long uptime.

## Background

The v0.32.0 review surfaced 22 issues (6 Critical, 9 Important, 7 Minor). v0.33.0 closed the 6 Critical (security + correctness). v0.34.0 closes the Important. The 7 Minor will roll into whatever modality-feature release touches the same files.

## Designed (2)

### A) Per-backend inference lock (#14)

**Problem:** Module-global `_inference_lock = Lock()` per modality means a worker hosting two embedders (e.g., MiniLM + Qwen3-Embedding) serializes every embed call through one lock, even though they have separate VRAM. Pre-v0.34.0 this was harmless because workers usually hosted one model per modality. With heavier multi-model workers it limits parallelism unnecessarily.

**Decision:** Each backend owns a `Lock`. Routes acquire it per call.

**Implementation:**
- `muse/core/registry.py`: in `ModalityRegistry.register`, attach `model._inference_lock = threading.Lock()` to every registered backend that does not already have one. Idempotent: skips reattach so backends that declare their own lock keep it.
- `muse/modalities/*/routes.py`: change `with _inference_lock:` to `with backend._inference_lock:` in every route. Drop the module-global `_inference_lock = Lock()`. Approximately 9 modules touched (audio_speech, audio_transcription, audio_embedding, audio_generation, image_generation, image_embedding, image_segmentation, image_upscale, embedding_text, video_generation, image_animation: those that currently use `_inference_lock`).
- Tests that build a `FakeModel` and register it through the registry get the lock attribute for free; no changes to existing test FakeModels needed.

**Out-of-scope tradeoff:** OOM under concurrent inferences on the same GPU stays visible (two diffusers pipelines peaking at 7GB each will collide on a 12GB card). This is intentional: the failure mode is loud and easy to diagnose. A future per-device lock (option B in brainstorming) is deferred until measured contention shows it matters.

### B) Admin enable atomicity (#7 + #8)

**Problem:** `admin/operations.enable_model` holds `state.lock` for the full 120s `wait_for_ready` window. Every other admin endpoint blocks during a slow GGUF load. The auto-restart monitor stalls if it tries to read `state.workers` mid-spawn. Releasing the lock during spawn opens a race where two concurrent enables for the same model spawn duplicate workers on different ports.

**Decision:** Hold the lock only across `state.workers` mutations. Spawn and wait happen outside the lock. Add a duplicate-spec guard so concurrent enables coalesce onto the first job.

**Implementation:**
1. Under `state.lock`:
   - Flip `set_enabled(model_id, True)` (catalog mutation, fast).
   - Walk `state.workers`: if any spec has `model_id == target` regardless of status (`pending`, `running`, `restarting`, `dead`), capture its `job_id` and return early. The caller polls the existing job rather than starting a new one. (γ-flavor idempotency.)
   - Else allocate `find_free_port()`, build `WorkerSpec(status="pending", job_id=new_job_id)`, `state.workers.append(spec)`.
2. Release lock.
3. Spawn worker (`spawn_worker(spec)`) and `wait_for_ready(spec, 120.0)` outside the lock.
4. Reacquire `state.lock` to flip `spec.status = "running"`. If `wait_for_ready` raised, flip to `dead` instead and let the monitor handle restart attempts (or not, depending on the failure).

**Read-side compatibility:** `/v1/admin/workers`, `/v1/admin/memory`, `/v1/admin/models/{id}/status`, and the auto-restart monitor all already filter by status (only `running` workers route requests, only failing `running` workers get restart attempts). A `pending` worker in `state.workers` is harmless to them. The gateway's `/v1/models` aggregation already skips non-running workers, so a `pending` model_id is invisible to clients until the spawn completes.

**JobStore interaction:** When the duplicate-spec branch returns the existing `job_id`, both callers see the same `JobStore` entry move through `pending → running → done` (or `failed`). The first caller's spawn drives. No JobStore API changes needed.

## Mechanical (5)

### C) Gateway AsyncClient leak on stream open failure (#9)

**File:** `src/muse/cli_impl/gateway.py`, lines around 284-292.

`httpx.AsyncClient(timeout=...)` is constructed, then `client.stream(...).__aenter__()` is awaited. If `__aenter__` raises (worker died mid-request, network blip), `client` leaks. Under load with flaky workers, file descriptor exhaustion eventually crashes the gateway.

**Fix:** wrap the `__aenter__` call in `try/except`; on failure `await client.aclose()` and re-raise as the existing failure path expects.

### D) Env-tunable size cap on segment + upscale (#11)

**Files:**
- `src/muse/modalities/image_generation/image_input.py`
- `src/muse/modalities/image_segmentation/routes.py`
- `src/muse/modalities/image_upscale/routes.py`

`decode_image_file`'s 10MB default is hardcoded; segment and upscale routes use it without an env tunable, while audio_transcription has `MUSE_ASR_MAX_MB`, audio_embedding has `MUSE_AUDIO_EMBEDDINGS_MAX_BYTES`, etc.

**Fix:** introduce `MUSE_IMAGE_INPUT_MAX_BYTES` (default `10 * 1024 * 1024`). `image_input._DEFAULT_MAX_BYTES` becomes a function `_default_max_bytes() -> int` that reads the env var per call. Segment and upscale routes accept the cap via the existing `decode_image_file(..., max_bytes=...)` signature.

### E) Operator-triggered restart resets counters (#12)

**File:** `src/muse/admin/routes/workers.py:60` (the `POST /v1/admin/workers/{port}/restart` handler).

After SIGTERM, the auto-restart monitor brings the worker back. But the original `restart_count` and `failure_count` carry over, so a worker that was 9 restarts deep into the `_MAX_RESTARTS=10` budget gets one more attempt and dies permanently. An operator's explicit restart should re-arm the budget.

**Fix:** in the restart endpoint, set `spec.restart_count = 0` and `spec.failure_count = 0` before SIGTERM. Document that the explicit restart is the way to recover a dead worker.

### F) audio_speech producer thread (#13)

**File:** `src/muse/modalities/audio_speech/routes.py:121`.

`loop.run_in_executor(None, _produce)` discards the returned `Future`. If the executor is saturated (rejects the schedule) or `_produce` raises before queueing, the consumer blocks on an empty queue forever. The audio_speech variant predates v0.33.0's chat_completion fix; chat now uses `threading.Thread(target=_producer, daemon=True).start()` for symmetry with the producer-thread pattern.

**Fix:** switch `audio_speech` to `threading.Thread(target=_produce, daemon=True).start()`. Drops the executor coupling, makes the lifecycle explicit, matches chat.

### G) Catalog purge cleans resolver weights (#15)

**File:** `src/muse/core/catalog.py`, around lines 500-522.

`remove(model_id, purge=True)` deletes the venv but leaves `~/.muse/weights/<local_dir>/` (the resolver-pulled weight cache). Pulling and removing 10 models accumulates 100+GB of orphan weights that the muse CLI cannot clean.

**Fix:** in the purge branch, if `entry.get("local_dir")` resolves under `_catalog_dir() / "weights"`, `shutil.rmtree` it. Bundled-pulled models leave their weights in HF's own cache (`~/.cache/huggingface`); muse does not own that, so we leave it alone (`huggingface-cli delete-cache` is the right tool there).

## Out of scope

- Per-device lock (option B from #14 brainstorming). Deferred until contention measurement shows it.
- Per-model locks for admin operations (option β from #7+#8 brainstorming). Deferred; α suffices.
- JobStore API changes for explicit coalescing (option γ in pure form). The duplicate-spec branch reuses the existing JobStore entry without API churn.
- Minor findings #16-#22. They roll into whichever modality release touches the same file next.

## Test strategy

**Designed (A, B):** focused regression tests.

- A: build a small `_LockingFake` model that records every call's lock identity, register two of them, hit them concurrently, assert each saw only its own lock (not a shared one).
- B: simulate a slow `wait_for_ready` (monkeypatch to `time.sleep(0.5)` then return), make the second concurrent enable observe the same `job_id`, assert it does NOT spawn a second worker.

**Mechanical (C-G):** one-test guards per fix.

- C: monkeypatch `httpx.AsyncClient.stream(...).__aenter__` to raise; assert `aclose` was called on the client.
- D: set `MUSE_IMAGE_INPUT_MAX_BYTES=12345`, call `decode_image_file` with a 13000-byte upload; assert ValueError mentions the env-derived cap.
- E: hit restart endpoint on a spec with `restart_count=5`; assert post-SIGTERM the spec has `restart_count == 0`.
- F: audio_speech streaming test that runs the full producer-consumer cycle; the existing test_streaming_yields_multiple_events_progressively already covers this if we just verify it still passes after the refactor.
- G: build a fake catalog entry with `local_dir = _catalog_dir()/"weights"/"foo"`, create the dir, call `remove(purge=True)`, assert the dir is gone.

**Full suite:** v0.33.0 left 2272 fast-lane tests passing. v0.34.0 should add roughly 7-10 new tests. Every pre-existing test must still pass; the per-backend lock change is the only one that could regression existing route tests, and the lock attribute is added before any route runs, so the behavioral surface is identical from the route's perspective (just faster under multi-model concurrency).

- F: `refactor(audio_speech): threading.Thread producer (matches chat)`
