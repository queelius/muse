# v0.40.0: Lazy Load + On-Demand LRU Eviction

**Status:** Spec, ready for implementation.
**Author:** Alex Towell
**Date:** 2026-05-05

## Goal (one sentence)

Replace muse's static "load all enabled models at boot" architecture with dynamic lazy loading: models load on first request, are evicted under memory pressure via LRU, and use live `pynvml` + `psutil` measurements to make memory decisions on shared GPU/CPU boxes.

## Motivation

Today, `muse serve` reads `~/.muse/catalog.json`, identifies every `enabled: true` model, and spawns a worker for each. Total memory consumption equals the sum of every enabled model's footprint. This caps the catalog at "what fits simultaneously," which is much smaller than what users actually want to have available. A user with a 12GB GPU might want to enable 20 models for a multi-modal workflow but can fit at most 3-4 of them concurrently.

The fix is to decouple "in service" (catalog `enabled`) from "currently loaded" (occupies a worker). Models in the catalog become a pool muse draws from on demand; the *currently loaded* set is sized to fit live free memory.

## Design summary

| Concept | v0.39.x (today) | v0.40.0 (this spec) |
|---|---|---|
| Boot behavior | Eager: spawn one worker per enabled model | Instant: zero workers, gateway healthy immediately |
| Loading | At supervisor boot, before gateway up | On first request to the model, after gateway up |
| Memory accounting | Implicit: sum-of-annotations vs total device memory | Live: `pynvml.nvmlDeviceGetMemoryInfo` + `psutil.virtual_memory` at decision time |
| Catalog capacity | Hard-capped by total device memory | Effectively unlimited; runtime LRU sizes the working set |
| First-request latency | ~free (model already loaded) | Cold-load cost (10-60s depending on model) |
| Multi-model coexistence | Static: declared at boot | Dynamic: as many fit live free memory minus headroom |

### Core mechanics

**Lazy load.** When a request arrives for a model that isn't currently loaded:

1. Look up the model's memory annotation (`capabilities.memory_gb`) or recorded probe peak (`measurements.<device>.peak_bytes`).
2. Query live free memory on the target device (GPU via `pynvml`, CPU via `psutil`).
3. Compute `available = live_free - headroom_margin`.
4. If `model.memory_gb <= available`: load directly via existing `enable_model` admin operation.
5. Else: enter eviction loop (see below).
6. Once loaded, route the request to the worker and increment the model's refcount.
7. After response, decrement refcount and update last-touched timestamp.

**On-demand LRU eviction.** When a load doesn't fit:

1. Identify eviction candidates: all currently-loaded models with refcount == 0 (no in-flight requests).
2. Sort candidates by last-touched timestamp ascending (least recently used first).
3. Evict the LRU candidate via existing `disable_model` admin operation. The OS reclaims the worker's memory on SIGTERM.
4. Poll free memory until either (a) it has increased enough to fit the requested model OR (b) ~2s timeout.
5. If `(a)`: load the requested model. If still doesn't fit and more candidates exist: repeat from step 2. If no more candidates and still doesn't fit: 503 `model_too_large_for_device`.

**Boot validation.** At supervisor startup:

1. Enumerate the catalog's `enabled: true` models.
2. For each, check if `memory_gb` annotation OR a `measurements.<device>.peak_bytes` exists.
3. If neither: mark `unservable_reason = "no memory estimate; run muse models probe"`.
4. If memory estimate exceeds device capacity at boot (live free + currently allocated): mark `unservable_reason = "exceeds device capacity"`.
5. Unservable models 503 immediately on request, before any load attempt.

**Probe becomes a sub-step of pull.** `muse pull <id>` runs probe at the end (after weights download + dep install) so newly-pulled models always have memory data. `--no-probe` flag opts out for cross-device pulls (e.g., pulling on a CPU box for later GPU deployment).

**Lazy-load passive observation.** Every cold load measures `free_before_load - free_after_load` and writes the delta back to `measurements.<device>.peak_bytes` if it exceeds the recorded value. Estimates self-heal upward toward reality.

### What stays the same

- Catalog schema (`enabled: true` keeps its meaning; new field `unservable_reason: str | null` is purely advisory).
- Admin REST API endpoints (no new ones; lazy load reuses `enable_model` and `disable_model`).
- HTTP wire compat (OpenAI-shape envelopes, all routes, all SDKs).
- Per-modality runtime classes (no changes).
- Per-venv worker model (each lazy load spawns one worker per model; co-located venv-group workers are a v0.41 optimization).

### What changes

- `muse serve` startup: instant; zero workers initially.
- First request per model: cold-load latency.
- `~/.muse/catalog.json`: gets a new top-level `last_loaded_at` per measurement entry, plus a writeback-on-observed-peak loop.
- `/v1/models`: gains `loaded: bool`, `last_loaded_at: iso8601 | null`, `unservable_reason: str | null` per model.
- `muse models list` glyphs become four-state for catalog rows: ● enabled+loaded, ◐ enabled+not-loaded, ○ disabled, ★/· recommended/available (unchanged).
- `muse models warmup <id>`: new CLI verb; pre-loads a model without serving traffic.
- Pre-flight probe ritual: `muse pull` runs probe at the end automatically.

## Components

### `LoadDirector` (new)

Lives in `src/muse/cli_impl/load_director.py`. Singleton on `SupervisorState`. Tracks:

```python
@dataclass
class LoadEntry:
    model_id: str
    worker_port: int
    memory_gb: float           # what we accounted for at load time
    refcount: int              # in-flight requests
    last_touched_at: float     # monotonic timestamp
    loaded_at: float

class LoadDirector:
    loaded: dict[str, LoadEntry]
    in_flight_loads: dict[str, asyncio.Future]  # singleton load coordination
    lock: threading.RLock                       # global load/evict critical section

    def acquire(self, model_id: str) -> int: ...        # returns worker_port
    def release(self, model_id: str) -> None: ...
    def warmup(self, model_id: str) -> int: ...         # like acquire but no refcount inc
    def observed_peak(self, model_id: str, peak_bytes: int) -> None: ...  # writeback
```

`acquire` is the hot path. It:
1. Holds the lock briefly to read `loaded`.
2. Increments refcount + updates `last_touched_at` if loaded.
3. If not loaded, releases lock, awaits `in_flight_loads[model_id]` if present, else creates the future and runs the load (with eviction loop) under the lock.
4. Returns the worker port.

The lock protects the *decision* (read live memory + decide eviction targets + record intent) but not the load *itself* (worker spawn happens outside, with the future tracked in `in_flight_loads`). This avoids serializing all loads behind one global lock.

### `MemoryProbe` (new)

Lives in `src/muse/core/memory_probe.py`. Wraps `pynvml` and `psutil` with graceful fallbacks.

```python
def gpu_free_gb(device_id: int = 0) -> float | None:

def cpu_free_gb() -> float:
    """Live RAM available."""

def init_pynvml() -> bool:
    """Idempotent. Returns False if init fails (no driver, AMD GPU, CPU-only host)."""
```

`pynvml` import is deferred (top-level `try / except` setting a sentinel `None`); if not installed at runtime, GPU loads fall back to bookkeeping with the declared budget if set, else 503.

### Gateway integration

`src/muse/cli_impl/gateway.py` route handler wraps each forwarded request:

```python
worker_port = director.acquire(model_id)
try:
    return await forward_to(worker_port, request)
finally:
    director.release(model_id)
```

`acquire` is the lazy-load entry point; the rest of the gateway is unchanged.

### Admin endpoint extensions

- `GET /v1/admin/memory` already exists; extend with `recent_decisions: [DecisionLogEntry, ...]` (last 20) showing load/evict events with timestamps, sizes, and reasons.
- New endpoint `POST /v1/admin/models/{id}/warmup` (synchronous wrapper that calls `LoadDirector.warmup`). The CLI's `muse models warmup` shells out via the admin client.

- New command `muse models warmup <id>` in `cli.py`. One-line wrapper around `AdminClient.warmup`.

## Data flow

### Cold load (model not loaded)

```
HTTP POST /v1/audio/speech model=kokoro-82m
  -> gateway.handler:
       worker_port = director.acquire("kokoro-82m")
         (1) decision phase, under lock:
              entry = loaded.get("kokoro-82m")  # None -> cold
              future = in_flight_loads.get("kokoro-82m")  # None
              future = asyncio.Future(); in_flight_loads["kokoro-82m"] = future
              memory_needed_gb = manifest.memory_gb  # 0.5
              free_before_gb = MemoryProbe.cpu_free_gb()  # 8.0
              available_gb = free_before_gb - headroom_cpu  # 8.0 - 2.0 = 6.0
              if memory_needed_gb > available_gb:
                  run eviction round (see Eviction below). May raise 503.
              # decision complete; lock released
         (2) load phase, OUTSIDE lock (long-running):
              port = enable_model("kokoro-82m")  # spawns worker, polls /health
              free_after_gb = MemoryProbe.cpu_free_gb()  # 7.5
              observed_consumed_bytes = (free_before_gb - free_after_gb) * 1024**3
         (3) commit phase, under lock again:
              loaded["kokoro-82m"] = LoadEntry(
                  worker_port=port, refcount=1,
                  memory_gb=memory_needed_gb,
                  last_touched_at=monotonic(), loaded_at=monotonic(),
              )
              del in_flight_loads["kokoro-82m"]
              future.set_result(port)
         (4) async writeback (no lock, fire-and-forget):
              if observed_consumed_bytes > recorded_peak_bytes:
                  catalog.measurements[device].peak_bytes = observed_consumed_bytes
       try:
           response = forward_to(port, request)
       finally:
           director.release("kokoro-82m")  # decrement refcount, update timestamp
       return response
```

The three-phase pattern (decide → load → commit) keeps the long-running load *outside* the lock so other models' decisions can proceed in parallel. Concurrent `acquire` calls for the *same* model see the future under the lock and await it instead of starting a second load. Concurrent `acquire` calls for *different* models can each go through the decision phase serially (fast) and load in parallel.

### Hot load (model already loaded)

```
director.acquire("kokoro-82m"):
  with lock:
      entry = loaded["kokoro-82m"]  # exists
      entry.refcount += 1
      entry.last_touched_at = monotonic()
      return entry.worker_port
```

Sub-millisecond.

### Concurrent first-requests for same model

Two requests arrive for unloaded model X simultaneously. First creates `in_flight_loads[X] = future`, drops the lock, starts the load. Second arrives, takes the lock, sees `in_flight_loads[X]` is present, releases lock, awaits the future. Both get the same worker port. One load happens, two requests served.

### Eviction round

```
director.acquire("sd-turbo"):  # 4GB needed, 1.5GB free
  with lock:
      memory_needed_gb = 4.0
      free_gb = MemoryProbe.gpu_free_gb()  # 1.5
      available_gb = 1.5 - 1.0 = 0.5  # headroom 1GB
      shortfall = 4.0 - 0.5 = 3.5

      candidates = [m for m in loaded.values()
                    if m.refcount == 0]
      candidates.sort(key=lambda e: e.last_touched_at)  # LRU first

      while shortfall > 0 and candidates:
          victim = candidates.pop(0)
          disable_model(victim.model_id)
          # poll until free memory increases or 2s timeout
          freed = wait_for_memory_release(victim.memory_gb, timeout=2.0)
          shortfall -= freed
          loaded.pop(victim.model_id, None)
          log "evicted X to make room for Y"

      if shortfall > 0:
          raise OperationError("model_too_large_for_device", status=503)

  # outside lock: spawn the new worker
  port = enable_model("sd-turbo")
  loaded["sd-turbo"] = LoadEntry(...)
  return port
```

## Error handling

- **No memory estimate AND no probe data**: 503 at request time with `unservable_reason: "no memory estimate; run muse models probe"`. Detected at boot; baked into the catalog state.
- **Memory estimate exceeds device capacity**: 503 at request time with `unservable_reason: "exceeds device capacity"`. Detected at boot.
- **Eviction can't free enough memory** (all loaded models have refcount > 0, OR sum of evictable memory < shortfall): 503 `model_too_large_for_device` with retry-after suggestion. Operator should wait for in-flight requests to finish.
- **Worker spawn fails (OOM at venv level despite the math saying it fits)**: cleaned up via existing supervisor monitor logic (worker won't pass /health, supervisor reports it dead). Translated to 503 with `worker_spawn_failed` code.
- **`pynvml` import / init failure**: log warning at boot. GPU loads use bookkeeping math against declared `MUSE_GPU_BUDGET_GB` if set; else mark all GPU models unservable.
- **`SIGTERM` doesn't release memory promptly**: poll-with-2s-timeout, then proceed regardless. If next load OOMs the worker, the spawn-fail path catches it.

## Configuration

```bash
# Optional declared cap (in addition to live measurement; muse uses min(declared, live))
MUSE_GPU_BUDGET_GB=11.0
MUSE_CPU_BUDGET_GB=24.0

# Headroom margins (subtracted from live free before deciding fit)
MUSE_GPU_HEADROOM_GB=1.0   # default
MUSE_CPU_HEADROOM_GB=2.0   # default

# Eviction tuning
MUSE_EVICT_RELEASE_TIMEOUT_SECONDS=2.0  # how long to wait for VRAM to free after SIGTERM
MUSE_EVICT_RELEASE_POLL_INTERVAL_SECONDS=0.05
```

No `MUSE_EAGER_LOAD` flag. Lazy is the only mode. Operators who really need eager can `muse models warmup <id>` for each model in their startup script.

## Testing strategy

### Unit tests (no subprocess, no real models)

- `tests/cli_impl/test_load_director.py`:
  - `acquire` on hot model: returns immediately, increments refcount, updates timestamp.
  - `acquire` on cold model with room: triggers load, returns port.
  - `acquire` on cold model with no room, evictable LRU candidates: triggers eviction in correct order.
  - `acquire` on cold model with no evictable candidates (all in-flight): 503 cleanly.
  - Concurrent `acquire` on same cold model: collapse to one load.
  - `release` decrements refcount; doesn't auto-evict (eviction is on-demand only in v0.40.0).
  - Boot validation: model with no annotation + no probe is unservable; model exceeding capacity is unservable.
- `tests/core/test_memory_probe.py`:
  - `gpu_free_gb` returns float when pynvml available; None when not.
  - `cpu_free_gb` always returns a float.
  - `init_pynvml` is idempotent and graceful on failure.

### Integration tests (in-process supervisor; mocked workers)

- Boot with empty load set, gateway healthy immediately.
- First request triggers load; second request to same model is hot.
- Two-model alternation under tight budget triggers swap; both eventually succeed.
- Three concurrent requests to three different unloaded models with budget for one: serial swap, all eventually serve.
- Disable while loaded: evicts immediately.
- Warmup: loads without traffic; subsequent request is hot.

### Slow e2e (real lightweight models)

- Pull a small model with `--no-probe`; first request 503s with the unservable reason. Manual `muse models probe` rescues. Verify message.
- Pull two small models that fit together; alternate requests; both stay loaded.
- Pull two models that don't fit together; alternate; verify swap behavior.

### Observability tests

- After a load, `/v1/admin/memory` shows the recent decision in `recent_decisions`.
- `muse models list` shows correct glyph state for loaded/unloaded.
- Structured log line emitted for every load + every eviction.

## Migration

Operators upgrading to v0.40.0 see:
- `muse serve` returns to prompt instantly instead of taking 5-30s for boot loads.
- First request to each model is slower than they're used to.
- `/v1/models` gains the `loaded` field; clients ignoring unknown fields are unaffected.
- `muse models list` shows `◐` for catalog-enabled-but-unloaded models.

No catalog edits are required. No CLI subcommands are removed. The change is additive at every public API.

For users who really want eager behavior: `muse serve & sleep 1; for m in $(muse models list --installed --json | jq -r '.[].id'); do muse models warmup $m; done` reproduces it exactly.

## Out of scope (deferred)

- **Co-located venv-group workers**: a worker hosting multiple sibling models in the same venv. v0.41 candidate; ~50MB Python interpreter savings per shared worker.
- **Idle-timeout eviction**: per-model `idle_timeout_seconds` for memory hygiene without traffic-driven pressure. v0.40.1 polish.
- **Multi-GPU support**: today muse uses device 0 only. Multi-GPU spreading + per-device budgets is a separate large feature.
- **Predictive pre-warm**: muse could observe traffic patterns and pre-load expected next-models. Out of scope; idle timeout + warmup verb cover the practical needs.
- **Cross-process memory tracking** beyond pynvml/psutil: cgroups, container memory limits, etc. v0.42+ if anyone needs it.

## Risks

- **Cold-start latency surprises users.** Mitigation: documented in CLAUDE.md and `/v1/models` exposes `loaded` so SDKs can warm up.
- **Eviction during high-concurrency thrashing.** Mitigation: refcount protection + 503 on uneviactable; operators see this in `/v1/admin/memory` decisions and can either declare a higher `MUSE_GPU_BUDGET_GB`, pre-warm critical models, or shed load.
- **`pynvml` driver-version mismatches** on operator boxes. Mitigation: graceful fallback to declared budget; clear warning at boot if pynvml unavailable.
- **`measurements` writeback corrupts under crash.** Mitigation: existing atomic write-then-rename for catalog.json applies (v0.30.0 work).
