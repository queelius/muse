# v0.35.0 Text Classifications Design

**Date:** 2026-05-03
**Status:** Approved for plan
**Closes:** #101 text/classification (zero-shot + sentiment)

## Goal

Add a `POST /v1/text/classifications` route on the existing `text/classification` modality, plus two bundled English models (sentiment + zero-shot NLI) and two curated multilingual variants. Coexists with the existing `/v1/moderations` route on the same modality tag.

## Why classifiers instead of LLMs

For non-AI-readers of this spec: a fine-tuned classifier is one forward pass through 125M-184M parameters and returns calibrated probabilities directly. An LLM doing the same task is hundreds of MB of VRAM, sampling tokens, and gives miscalibrated probabilities at best. Classifiers win on latency (5-15ms CPU vs hundreds of ms LLM), on calibration (real probabilities for thresholding/ensembling), and on auditability (deterministic given fixed weights). The right deployment pattern is classifiers for hot paths and audit-critical decisions, LLMs for the long tail and reasoning-over-input cases. Muse hosts both for the same reason a real ML stack does.

## Wire shape

`POST /v1/text/classifications`:

Request:

```json
{
  "model": "twitter-roberta-base-sentiment-latest",
  "input": "I love this!",
  "top_k": 3,
  "candidate_labels": null,
  "multi_label": false
}
```

`input` is `str | list[str]`. `candidate_labels` is `list[str] | None`; presence triggers zero-shot dispatch. `top_k` is `int | None` (None = return all labels). `multi_label` is `bool` (zero-shot only); when True, each label scored independently rather than via softmax.

Response:

```json
{
  "id": "classify-...",
  "model": "twitter-roberta-base-sentiment-latest",
  "results": [
    [
      {"label": "positive", "score": 0.94},
      {"label": "neutral", "score": 0.04},
      {"label": "negative", "score": 0.02}
    ]
  ]
}
```

`results` is always a list of lists (one inner list per input), even when `input` is a single string. Inner list is sorted by score descending. Uniform whether the input is sentiment or zero-shot.

## Capability gating

Runtime dispatch is decided by manifest capabilities on the loaded model:

- `supports_classification: True` (fine-tuned classifier head). Route accepts requests without `candidate_labels`.
- `supports_zero_shot: True` (NLI head). Route accepts requests with `candidate_labels`.

Both can be True if a model genuinely supports both heads (none of our bundled or curated do; flag exists for future custom models).

Mismatch returns 400:

- Request has `candidate_labels` but model has `supports_zero_shot != True`: `400 zero_shot_not_supported`.
- Request has no `candidate_labels` and model has `supports_classification != True`: `400 candidate_labels_required`.

This mirrors the `supports_img2img` / `supports_inpainting` pattern from image_generation.

## Runtime architecture

Two runtimes coexist on the modality:

1. **`HFTextClassifier`** (existing, untouched). Wraps `AutoModelForSequenceClassification`. Handles `/v1/moderations` and `/v1/text/classifications` for fine-tuned classifiers. Returns full label distribution; the moderations route reduces to flag/score, the new route returns the list.

2. **`HFZeroShotPipeline`** (new). Wraps `transformers.pipeline("zero-shot-classification")`. The pipeline does the NLI premise/hypothesis construction for free; runtime body is approximately 80 LOC. New file: `src/muse/modalities/text_classification/runtimes/hf_zero_shot.py`.

The HF plugin sniffs by tag:

- `zero-shot-classification` tag → resolve to `HFZeroShotPipeline`.
- `text-classification` tag (any other) → resolve to `HFTextClassifier` (existing).

Sniff order matters: zero-shot wins because some models carry both tags.

## Bundled models (2)

1. **`twitter-roberta-base-sentiment-latest`** (English, 125M, 3-label: positive/neutral/negative).
   - `hf_repo`: `cardiffnlp/twitter-roberta-base-sentiment-latest`
   - License: MIT
   - Size: ~500MB
   - `capabilities`: `{supports_classification: True, supports_zero_shot: False, memory_gb: 0.6, device: cpu}`
   - Bundled script: `src/muse/models/twitter_roberta_base_sentiment_latest.py`. Reuses `HFTextClassifier`. Joins the CI smoke matrix.

2. **`deberta-v3-base-zeroshot-v2.0`** (English, 184M, NLI head).
   - `hf_repo`: `MoritzLaurer/deberta-v3-base-zeroshot-v2.0`
   - License: MIT
   - Size: ~770MB
   - `capabilities`: `{supports_classification: False, supports_zero_shot: True, memory_gb: 1.2, device: cpu}`
   - Bundled script: `src/muse/models/deberta_v3_base_zeroshot_v2_0.py`. Uses new `HFZeroShotPipeline`. Joins the CI smoke matrix.

## Out of scope

- Topic/intent classifiers (`tweet-topic-21-multi`, etc.). User declined; the route already serves them via `muse pull hf://...` without a curated alias.
- Multi-input batching beyond what the pipeline handles natively.
- Few-shot example support (Cohere classify shape). Our HF model lineup is fine-tuned, not few-shot learners.
- Training utilities. Muse is inference-only.

## Test strategy

**Protocol + codec tests:** unit-test the request/response dataclasses, the `top_k` truncation, the `candidate_labels`-presence dispatch.

**Route tests:** FakeModel that satisfies the protocol; assert capability-gate 400s, top_k truncation, single-vs-list input shape uniformity, OpenAI error envelope on backend failure.

**Runtime tests:** `HFZeroShotPipeline` with a mocked `transformers.pipeline` returning canned `{labels, scores}` output. Verify multi_label flag forwards. Verify lazy-import discipline (`muse --help` works without transformers).

**Bundled-script tests:** mocked transformers; test `Model.__init__`, capability declaration, classify/zero-shot method shape.

**HF plugin tests:** sniff dispatches zero-shot-classification tag → HFZeroShotPipeline; text-classification tag → HFTextClassifier.

**Slow e2e:** in-process supervisor + bundled sentiment + a short input. Skip if heavy weights aren't downloadable in CI.

**Smoke matrix:** add both bundled to `.github/workflows/fresh-venv-smoke.yml`. Both are <200M params and CPU-friendly.
