# v0.44.0 TRELLIS Image-to-3D: Design

**Status:** Approved (pending user review of this spec)
**Target version:** v0.44.0
**Task:** #108 (3D generation runtime classes), Phase 2 of 4 (TRELLIS in place of Wonder3D)
**Date:** 2026-05-11

## Goal

Add a `TRELLISRuntime` to the existing `3d/generation` modality so the curated TRELLIS entry actually works end-to-end. The runtime wraps the official TRELLIS SDK loaded via `trust_remote_code=True` (already supported by the family registry from v0.43.2). Fix a latent curated-entry bug where the entry claimed `supports_text_to_3d: true` for an image-only model URI. Drop the previously-planned Wonder3D phase (deferred indefinitely) in favor of jumping directly to TRELLIS.

## Context

v0.41.0 shipped four curated 3D entries (TripoSR, Wonder3D, TRELLIS, Hunyuan3D-2). v0.43.0 through v0.43.3 wired the per-family dispatcher and shipped the Shap-E runtime. This release plugs TRELLIS into the now-locked registry pattern.

Wonder3D (originally planned as v0.44.0) is deferred. Its multi-view diffusion plus NeuS reconstruction architecture does not fit muse's "one pipeline call" runtime pattern cleanly; the curated quality of TRELLIS exceeds Wonder3D's; and Wonder3D's CC-BY-NC-4.0 license narrows audience. The Wonder3D curated entry remains in `curated.yaml` with a "deferred indefinitely" annotation; revisit if demand emerges.

## Architecture

### TRELLISRuntime

```python
# src/muse/modalities/model_3d_generation/runtimes/trellis.py

class TRELLISRuntime:
    """Image-to-3D via Microsoft's TRELLIS SDK.

    Wraps the official TRELLIS pipeline loaded via transformers'
    AutoPipeline.from_pretrained with trust_remote_code=True. The SDK
    code is published in the TRELLIS HuggingFace repo and downloaded
    along with weights on muse pull. Returns GLB bytes through trimesh
    export.
    """

    model_id: str
    supports_image_to_3d: bool = True
    supports_text_to_3d: bool = False

    def __init__(
        self,
        *,
        model_id: str,
        hf_repo: str,
        local_dir: str | None = None,
        device: str = "auto",
        dtype: str = "fp16",
        trust_remote_code: bool = True,
        seed: int | None = None,
        sparse_structure_steps: int = 12,
        slat_steps: int = 12,
        **_: Any,
    ) -> None: ...

    def image_to_3d(
        self, image: Any, **kwargs: Any,
    ) -> list[Generation3DResult]: ...

    def text_to_3d(
        self, prompt: str, **kwargs: Any,
    ) -> list[Generation3DResult]:
        raise NotImplementedError(
            "TRELLIS-image-large is image-only. The TRELLIS text-to-3D "
            "variant has not been curated; pull a text-capable TRELLIS "
            "variant (when available) and register it separately."
        )
```

### Runtime details

The TRELLIS pipeline call shape (per the official repo's example notebooks):

```python
# Approximate; actual API verified at implementation time against
# the repo's custom code that arrives with trust_remote_code.
pipeline = SomeAutoLoader.from_pretrained(
    "JeffreyXiang/TRELLIS-image-large",
    trust_remote_code=True,
    torch_dtype=torch.float16,
)
pipeline.to("cuda")
outputs = pipeline.run(
    image,
    seed=seed,
    sparse_structure_sampler_params={"steps": 12},
    slat_sampler_params={"steps": 12},
)
mesh = outputs["mesh"][0]  # or similar; verified at implementation time
glb_bytes = mesh.export(file_type="glb")
```

The implementation step will verify the exact loader class name and output shape against the TRELLIS repo's `transformers_modules/<repo>/pipeline.py` (which becomes available after the first pull with `trust_remote_code=True`). The plan's code blocks will be marked as best-effort approximations, and the implementer should consult the real downloaded SDK during implementation rather than trusting the speculative shape.

This is the same lesson as v0.43.0 Shap-E: trust the actual API, not the implementer's mental model. The runtime test mocks must match the real pipeline contract, not a guessed one.

### Family registry entry

New entry appended to `_FAMILIES` in `model_3d_generation/hf.py`:

```python
_TRELLIS_RUNTIME_PATH = (
    "muse.modalities.model_3d_generation.runtimes.trellis:TRELLISRuntime"
)
_TRELLIS_PIP_EXTRAS = (
    "torch>=2.1.0",
    "transformers>=4.46.0",
    "diffusers>=0.27.0",
    "trimesh",
    "accelerate",
    # TRELLIS's SDK code in the HF repo imports additional packages
    # (xformers, kaolin, etc.); the implementer verifies the full pip
    # requirements list against transformers_modules/<repo>/requirements.txt
    # or equivalent in the snapshot at implementation time.
)


# Append to _FAMILIES:
_Family(
    name_hints=("trellis",),
    runtime_path=_TRELLIS_RUNTIME_PATH,
    pip_extras=_TRELLIS_PIP_EXTRAS,
    capability_overrides={
        "supports_image_to_3d": True,
        "supports_text_to_3d": False,
    },
    trust_remote_code=True,
),
```

The `trust_remote_code=True` flag flows through `_resolve()` into the synthesized manifest's `capabilities` dict (per v0.43.2's `if family.trust_remote_code:` branch). The TRELLISRuntime constructor reads it from manifest capabilities via the standard `**_` absorption.

### Curated YAML fix

  modality: 3d/generation
  size_gb: 1.5
  description: "TRELLIS image+text to 3D: ~1.5GB at fp16, dual-direction (image and text), requires trust_remote_code, MIT (runtime arrives in v0.45.0)"

    memory_gb: 6.0
    device: cuda
    supports_image_to_3d: true
    supports_text_to_3d: true

Has two problems: (1) the id name "trellis-text-image" implies dual-direction, but the URI is image-only; (2) `supports_text_to_3d: true` is a capability lie. Both fixed:

## Error handling

- Missing torch / transformers / trimesh: raises RuntimeError with actionable `muse models refresh trellis-image` hint (mirrors Shap-E and other runtimes).
- `text_to_3d` called: raises NotImplementedError with a message pointing at the image-only nature of TRELLIS-image-large.
- Pipeline load failure (network, corrupted snapshot): exception propagates; supervisor's worker spawns a fresh process on next request (existing pattern).
- Generation failure (CUDA OOM, malformed input): exception propagates to the route layer (500 response, existing pattern).

## Migration / breaking changes

**One breaking change.** The curated entry id `trellis-text-image` becomes `trellis-image`. Anyone who pulled the original entry under the old id would need to re-pull under the new id. Mitigations:

- The original entry never worked end-to-end (capability lie plus no runtime), so no real users are broken.
- The rename eliminates the capability lie; the new id is honest.
- Release notes call this out explicitly with the upgrade instruction (`muse models remove trellis-text-image && muse pull trellis-image`).

The Wonder3D curated entry stays but is annotated deferred-indefinitely. No id change.

The Hunyuan3D-2 entry stays unchanged but its description's "runtime arrives in" version is updated.

Catalog format unchanged.

## Risk register

- **Pipeline class name uncertainty.** The exact loader (`TrellisImageTo3DPipeline`, `AutoPipeline`, or a custom class registered via `trust_remote_code`) is not confirmed at spec time. The implementer will verify against the actual downloaded TRELLIS code during implementation. If the actual API diverges from the spec's speculative shape, the runtime adapts.
- **TRELLIS pip dependencies.** The SDK may pull in xformers, kaolin, or similar packages that have CUDA-version-specific wheels. The implementer verifies the requirements list against the snapshot at implementation time and updates `_TRELLIS_PIP_EXTRAS` accordingly.
- **GLB export shape.** Whether the SDK returns a `trimesh.Trimesh` directly, raw vertex/face arrays, or its own mesh type is verified at implementation time. The runtime adapts to the actual return type.

These risks are the same pattern that v0.43.0 Shap-E hit (where the speculative API differed from the real one and was caught by post-release code review). The lesson from v0.43.1: verify the real API at implementation time, not the speculative one.
