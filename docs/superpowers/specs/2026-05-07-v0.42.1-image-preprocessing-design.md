# v0.42.1 Cross-Modality Image Preprocessing: Design

**Status:** Approved (pending user review of this spec)
**Target version:** v0.42.1
**Task:** #236 (cross-modality consolidation, dormant since v0.41.2 design review)
**Date:** 2026-05-07

## Goal

Lift the generic image-processor fallback machinery from `muse.modalities.image_ocr.runtimes.hf_vision2seq` into a shared `muse.core.image_preprocessing` module. Add a manifest-driven override hatch (`capabilities.image_processor_overrides`). Drop the silently-wrong ViT-defaults fallback tier. Migrate only `image_ocr` in v0.42.1; defer `image_embedding` / `image_cv` / `image_segmentation` adoption until a real second caller emerges.

## Context

The v0.41.2 TexTeller fix introduced three private helpers in `image_ocr/runtimes/hf_vision2seq.py`:
- `_read_encoder_preprocess_hints(src)`: reads `config.json` for encoder preprocessing hints.
- `_DerivedImageProcessor`: synthesizes a minimal preprocessor from those hints.
- `_build_image_processor_fallback`: three-tier dispatch (AutoImageProcessor, derived, ViT defaults).

The v0.41.2 design reviewer flagged this as the right behavior in the wrong location: the helpers are generic; the dispatch is modality-specific. The reviewer also flagged that tier 3 (ViT defaults) is silently wrong for the very class of repos this fallback targets, and recommended a manifest-driven override hatch.

This spec implements the minimal lift: move helpers, add overrides, drop tier 3.

## Decisions (locked)

| # | Question | Choice |
|---|---|---|
| Q1 | Migration scope | Minimal: lift to `muse.core.image_preprocessing`, migrate only `image_ocr`. Other modalities keep current behavior. |
| Q2 | `image_processor_overrides` schema | Encoder-config fields only: `{num_channels, image_size, image_mean?, image_std?}`. Override-first dispatch (skip AutoImageProcessor entirely when set). |
| Q3 | Tier 3 (ViT defaults) | Drop entirely. Replace with a structured `ImageProcessorError` whose message points operators at the override hatch. |
| Q4 | Module name | `muse.core.image_preprocessing` |
| Q5 | Class name | `DerivedImageProcessor` (public, no underscore prefix) |

## Architecture

### New module: `muse.core.image_preprocessing`

Public API surface:

```python
def read_encoder_hints(src: str) -> dict:
    """Read encoder preprocessing hints from config.json.

    Returns a dict containing any subset of keys: num_channels,
    image_size, image_mean, image_std. Empty dict on missing or
    malformed config.json.
    """


class DerivedImageProcessor:
    """Synthesize a minimal preprocessor from explicit hyperparams.

    Mimics the AutoImageProcessor interface for callers: callable
    with (image, return_tensors="pt"), returns a BatchFeature with
    pixel_values shape (1, num_channels, H, W) supporting .to(device).
    """
    def __init__(
        self,
        *,
        num_channels: int = 3,
        image_size: int | tuple[int, int] = 224,
        image_mean: list[float] | None = None,
        image_std: list[float] | None = None,
    ) -> None: ...

    def __call__(self, image, *, return_tensors: str = "pt"): ...


def build_image_processor(
    src: str,
    *,
    overrides: dict | None = None,
    model_id: str,
) -> Any:
    """Three-tier dispatch:

    1. If `overrides` is non-empty: skip AutoImageProcessor; build
       DerivedImageProcessor(**overrides) directly. Operator-trust path.
    2. Else: try AutoImageProcessor.from_pretrained(src). Return on success.
    3. On AutoImageProcessor failure: read encoder hints from config.json.
       If hints found, build DerivedImageProcessor(**hints).
    4. Else: raise ImageProcessorError pointing at the override hatch.
    """


class ImageProcessorError(RuntimeError):
    """Raised when the dispatch ladder is exhausted without a usable
    processor. Message includes the override-hatch hint."""
```

The new module has no muse-internal dependencies beyond `transformers`/`torch`/`PIL` (lazy-imported inside the helpers, same pattern as runtimes). It exports the four names above.

### Image-OCR migration

`src/muse/modalities/image_ocr/runtimes/hf_vision2seq.py` changes:

1. Drop the three private helpers (`_read_encoder_preprocess_hints`, `_DerivedImageProcessor`, `_build_image_processor_fallback`).
2. Drop the `_SimpleBatch` shim (already gone in v0.41.2 cleanup).
3. Import from the new module: `from muse.core.image_preprocessing import build_image_processor, ImageProcessorError`.
4. Constructor accepts new kwarg `image_processor_overrides: dict | None = None` (absorbed by `**_` if not declared in older manifests).
5. Replace the existing fallback call with:
   ```python
   self._image_processor = build_image_processor(
       src, overrides=image_processor_overrides, model_id=model_id,
   )
   ```

### Manifest schema

New optional capability:

```yaml
capabilities:
  image_processor_overrides:
    num_channels: 1            # required when overrides is set
    image_size: 448            # int OR [height, width]
    image_mean: [0.5]          # optional; defaults to [0.5] * num_channels
    image_std: [0.5]           # optional; defaults to [0.5] * num_channels
```

Validation at manifest-load time (NOT at first request): if `image_mean` is set and its length does not match `num_channels`, raise. Same for `image_std`. Caught by:

- Bundled scripts: validation runs in their MANIFEST validation tests.
- Curated YAML: validation runs in `find_curated()` / curated-load tests.
- Resolver-synthesized: never sets overrides, so no validation needed at the resolver.

The override fields flow through the existing `manifest.capabilities` -> runtime constructor splat. No registry or catalog schema changes.

### Curated TexTeller update

Add `image_processor_overrides` to the existing `texteller` curated entry:

```yaml
- id: texteller
  description: TexTeller (...).
  uri: hf://OleehyO/TexTeller
  modality: image/ocr
  capabilities:
    memory_gb: 2.0
    device: cpu
    image_processor_overrides:
      num_channels: 1
      image_size: 448
```

After this change, TexTeller no longer relies on the runtime sniffing `config.json` at first request. The manifest declares ground truth, robust to transformers updates that might shift AutoImageProcessor behavior.

### Tier 3 removal: error message

When AutoImageProcessor fails AND no encoder hints are available, raise:

```
ImageProcessorError(
  f"Cannot load image processor for {src!r}: AutoImageProcessor "
  f"failed and config.json provided no usable encoder hints "
  f"(checked: num_channels, image_size). Add "
  f"capabilities.image_processor_overrides in the manifest with "
  f"explicit values, e.g. {{num_channels: 1, image_size: 448}}."
)
```

The route layer translates this to a 500 (server-side configuration error, not a 400 user error). For CPU smoke tests, the supervisor would log the exception clearly.

## Out of scope for v0.42.1

- `image_embedding` migration. Its fallback ladder is `AutoProcessor -> AutoFeatureExtractor`, addressing a different gap (no `processor_config.json` rather than no `preprocessor_config.json`). The new module's `build_image_processor` does not subsume that path. A future task can either generalize the lifted module or migrate image_embedding's `_load_processor` directly.
- `image_cv` adoption (depth/keypoint/object_detection runtimes). No user-reported failure; YAGNI.
- `image_segmentation` adoption. SAM2 has its own loader; not a candidate for the fallback ladder.
- Upstream transformers issue or PR (the design reviewer's recommendation #4). Track separately as a community contribution.

## Testing strategy

**Unit tests (fast lane):** ~15 tests in the new `test_image_preprocessing.py`. Test the four-tier dispatch matrix end-to-end with mocked AutoImageProcessor / config.json files.

**Regression watchdogs:**
- The existing `test_hf_vision2seq.py` constructor-integration tests (specifically `test_constructor_split_path_uses_derived_when_auto_image_processor_fails`) must continue to pass after migration. They verify the runtime still wires the fallback correctly when AutoImageProcessor fails.
- The TexTeller live-probe (manual: `muse models probe texteller`) must succeed both before and after the migration.

**Coverage of the override hatch:** The bundled and curated entries are the main consumers. The texteller curated entry's override block has its own validation test.

## Migration / breaking changes

None. Text-only flows are unaffected. image_ocr behavior is unchanged for repos that ship `preprocessor_config.json` (tier 2 succeeds). For repos that hit tier 3 today (encoder-config-derived), behavior is unchanged unless the curated entry adopts the override hatch (in which case the operator path is more explicit).

The only behavioral change: removing tier 3 (ViT defaults). Repos that previously fell through to ViT defaults will now raise `ImageProcessorError` instead of silently returning a wrong-shape processor. This is intentional. Operators adopting muse for new VLM/OCR repos hit a clear error pointing at the override hatch, instead of garbage outputs from a mismatched processor. The bundled `trocr-base-printed` model ships with `preprocessor_config.json` so it never reaches tier 3. The bundled `nougat` (curated, not bundled) similarly ships `preprocessor_config.json`. The curated `texteller` is the one entry that benefits from the override; its update is part of this release.

## Release plan

`0.42.0 -> 0.42.1` (patch bump). New core module + manifest field is additive; no public API breaks. Migration of image_ocr is an internal refactor.

Ritual mirrors v0.41.2 / v0.42.0:
1. Bump pyproject.
2. Run fast lane (expect 2996 + ~15 new = ~3011 tests passing).
3. Live-verify: `muse models probe texteller` still succeeds.
4. Commit, tag `v0.42.1`, push, GitHub release.

## Future work (separate releases)

- `muse.core.image_preprocessing.build_image_processor` could be extended with more dispatch tiers (e.g. AutoFeatureExtractor for image_embedding's gap). When that becomes desirable, the API absorbs it without breaking existing callers.
- A `capabilities.image_processor_overrides` could grow optional fields beyond the four core ones (`do_pad`, `do_center_crop`, `resample`, etc.). Add as needed; the validation layer ignores unknown fields today.
- Migrate other modalities to consume `build_image_processor` once a second caller has a clear motivating failure. The lift is in place; adoption is independent per modality.
