# v0.43.0 Shap-E Runtime + Per-Family Dispatch: Design

**Status:** Approved (pending user review of this spec)
**Target version:** v0.43.0
**Task:** #108 (3D generation runtime classes), Phase 1 of 4 (Shap-E first)
**Date:** 2026-05-09

## Goal

Add a `ShapERuntime` to the existing `3d/generation` modality so the curated `shap-e` entry actually works end-to-end. Refactor the 3D HF resolver to dispatch per-family backend_path, providing the extension point for future TRELLIS / Wonder3D / Hunyuan3D-2 runtimes (v0.44.0 through v0.46.0). Fix v0.41.0's documentation/runtime gap where curated entries pointed at TripoSR despite needing different loaders.

## Context

v0.41.0 shipped the `3d/generation` modality with `TripoSRRuntime` and four curated entries: `trellis-text-image`, `wonder3d-base`, `hunyuan3d-2`, `shap-e`. The HF resolver's `_resolve()` always sets `backend_path = TripoSRRuntime`, regardless of repo family. Result: curated entries persist TripoSR as their runtime in `catalog.json`, but TRELLIS/Wonder3D/Hunyuan3D/Shap-E weights are not loadable by TripoSR. First probe / inference fails for any of these four curated models. This spec begins addressing that debt by adding the simplest of the four runtimes (Shap-E, ~600MB, OpenAI text-to-3D) and the dispatch infrastructure that future runtimes plug into.

## Architecture

### ShapERuntime

```python
# src/muse/modalities/model_3d_generation/runtimes/shape_e.py

class ShapERuntime:
    """Generic Shap-E runtime over diffusers.ShapEPipeline.

    Wraps `diffusers.ShapEPipeline` for text-to-3D generation. Returns
    GLB-encoded mesh output through the existing model_3d_generation
    codec.
    """

    model_id: str
    supports_text_to_3d: bool = True
    supports_image_to_3d: bool = False

    def __init__(
        self,
        *,
        model_id: str,
        hf_repo: str,
        local_dir: str | None = None,
        device: str = "auto",
        dtype: str = "fp16",
        guidance_scale: float = 15.0,
        num_inference_steps: int = 64,
        frame_size: int = 256,
        **_: Any,
    ) -> None: ...

    def generate_text_to_3d(
        self, prompt: str, **kwargs: Any,
    ) -> Generation3DResult: ...

    def generate_image_to_3d(
        self, image: Any, **kwargs: Any,
    ) -> Generation3DResult:
        raise NotImplementedError(
            "Shap-E base is text-to-3D only; image-to-3D would require "
            "the separate openai/shap-e-img2img variant which is not "
            "supported in v0.43.0."
        )
```

The runtime follows muse's deferred-imports pattern: torch, diffusers, trimesh stay as module-level sentinels until `_ensure_deps()` lazy-imports them. Tests patch sentinels directly. `_ensure_deps` short-circuits on non-None.

### Per-family resolver dispatch

The 3D HF resolver's `_resolve()` becomes a per-family dispatcher. The constants and dispatch helper:

```python
# In model_3d_generation/hf.py, near the top

_TRIPOSR_RUNTIME_PATH = "muse.modalities.model_3d_generation.runtimes.triposr:TripoSRRuntime"
_SHAPE_E_RUNTIME_PATH = "muse.modalities.model_3d_generation.runtimes.shape_e:ShapERuntime"

_TRIPOSR_PIP_EXTRAS = ("torch>=2.1.0", "tsr", "trimesh", "Pillow")
_SHAPE_E_PIP_EXTRAS = (
    "torch>=2.1.0",
    "diffusers>=0.27.0",
    "transformers",
    "trimesh",
)


def _runtime_path_for(repo_id: str) -> str:
    """Pick the right runtime backend for an HF repo by family.

    Extension point: when v0.44.0 adds Wonder3D, add one branch here
    plus one runtime file. Curated entries get their runtime via this
    dispatch; ad-hoc HF URI pulls fall through to TripoSR.
    """
    name = repo_id.lower()
    if "shap-e" in name or "shape-e" in name:
        return _SHAPE_E_RUNTIME_PATH
    return _TRIPOSR_RUNTIME_PATH


def _pip_extras_for(runtime_path: str) -> tuple[str, ...]:
    if runtime_path == _SHAPE_E_RUNTIME_PATH:
        return _SHAPE_E_PIP_EXTRAS
    return _TRIPOSR_PIP_EXTRAS
```

`_resolve()` consults `_runtime_path_for(repo_id)` and `_pip_extras_for(runtime_path)` to populate the resolved manifest.

This shape is extension-clean: each future runtime release (v0.44.0 / v0.45.0 / v0.46.0) adds one constant plus one dispatch branch. No restructuring needed.

### Capability surface

Shap-E is text-to-3D only. The runtime:
- Sets `supports_text_to_3d = True` and `supports_image_to_3d = False` as class-level attributes (mirrors how TripoSRRuntime / HFVisionLanguageModel surface capability flags).
- Receives manifest capabilities through the constructor's `**_` absorption (the catalog's runtime instantiation splats capabilities into kwargs). Capability checks at the route layer use `registry.manifest(MODALITY, model_id).get("capabilities", {})` per the v0.42.0 precedent established in chat_completion routes.

### Output handling

`diffusers.ShapEPipeline(...)` returns an object with `.meshes` (list of Mesh objects). Each Mesh has `vertices` (N x 3 float array) and `faces` (M x 3 int array). The runtime adapts via:

```python
import trimesh
mesh_data = result.meshes[0]
mesh = trimesh.Trimesh(
    vertices=mesh_data.vertices.cpu().numpy(),
    faces=mesh_data.faces.cpu().numpy(),
)
glb_bytes = mesh.export(file_type="glb")
```

The existing `model_3d_generation/codec.py` already handles GLB bytes; no codec changes.

## Capability schema

No new manifest fields. The existing `capabilities.supports_text_to_3d` / `supports_image_to_3d` / `output_format` from v0.41.0 are sufficient.

The Shap-E curated entry already has these set correctly:

```yaml
- id: shap-e
  uri: hf://openai/shap-e
  modality: 3d/generation
  capabilities:
    memory_gb: 2.0
    device: cuda
    supports_image_to_3d: false
    supports_text_to_3d: true
    output_format: glb
```

After v0.43.0 ships, the resolver picks `_SHAPE_E_RUNTIME_PATH` for this URI, the catalog persists it, and `muse pull shap-e` produces a working entry.

## Error handling

- Missing torch: constructor raises `RuntimeError` with `muse models refresh shap-e` hint (mirrors TripoSR / VLM patterns).
- Missing diffusers: same pattern, message names `diffusers>=0.27.0`.
- Missing trimesh: same pattern.
- `generate_image_to_3d` called: `NotImplementedError` with message explaining Shap-E base is text-only and pointing at the route layer's capability check (which should have rejected the request before reaching the runtime).
- Generation failure (CUDA OOM, malformed prompt): exception propagates to the route layer, which returns 500 (existing pattern).

## Migration / breaking changes

None.

- TripoSR continues to work unchanged; the resolver's default fallback for ad-hoc URIs is still TripoSR.
- The TRELLIS / Wonder3D / Hunyuan3D-2 curated entries continue to resolve to TripoSR backend, which means `muse pull trellis-text-image` etc. continue to NOT work end-to-end. The curated description is updated to call this out, framing it as future work rather than a regression.
- Catalog format unchanged; new manifests just have a different `backend_path` (Shap-E URIs only).
- Wire format unchanged; the route protocol is not affected.

## Out of scope for v0.43.0

- TRELLIS / Wonder3D / Hunyuan3D-2 runtimes. Each gets its own release (v0.44.0 / v0.45.0 / v0.46.0) with the same dispatch pattern (one constant + one dispatch branch in `_runtime_path_for` per family, plus the runtime file).
- Shap-E radiance-field output. The route declares `output_format: glb`; the runtime produces meshes and exports GLB. The radiance-field path would need codec extension and is not requested.
- Shap-E image-to-3D. The `openai/shap-e-img2img` variant exists but is not curated; v0.43.0 ships text-to-3D Shap-E base only.
- Bundling Shap-E. Bundled models in muse must be CPU-friendly; Shap-E requires GPU.
