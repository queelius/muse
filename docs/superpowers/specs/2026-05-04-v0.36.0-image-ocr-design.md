# v0.36.0 image/ocr Design

**Date:** 2026-05-04
**Status:** Approved for plan
**Closes:** #106 image/ocr (TrOCR / Donut / document-QA)

## Goal

A new `image/ocr` modality with one route `POST /v1/images/ocr`, image-in (multipart) and text-out. Bundles `trocr-base-printed` only; curates Nougat, TexTeller, and TrOCR-handwritten. Donut / DocVQA / structured document parsing are out of scope; they may land later as a separate `image/document` modality if demand justifies.

## Why this scope

OCR is three distinct tasks: plain text extraction, structured document parsing (Donut JSON), and document QA (image + question). They share the vision-encoder + text-decoder shape but diverge in wire shape and use case. v1 covers only plain OCR. Math/LaTeX is included via the Nougat and TexTeller curated entries, since the wire shape and runtime architecture handle them naturally (same image-to-text contract).

## Wire shape

`POST /v1/images/ocr` accepts `multipart/form-data`:

| field | type | required | notes |
|---|---|---|---|
| `image` | file | yes | PNG, JPEG, or WebP. Subject to `MUSE_IMAGE_INPUT_MAX_BYTES` cap (default 10MB). |
| `model` | str | no | Catalog id; falls back to modality default. |
| `prompt` | str | no | Task hint. Used by Nougat (`<s_doc>` etc.); ignored by TrOCR. |
| `max_new_tokens` | int | no | Caps generation length. Default 512; per-model override via `capabilities.max_new_tokens`. Hard ceiling 4096. |

Response (JSON):

```json
{
  "id": "ocr-<uuid>",
  "model": "trocr-base-printed",
  "text": "extracted text",
  "usage": {"completion_tokens": 47}
}
```

Mirrors `/v1/audio/transcriptions` for symmetry: one file in, text out, usage block for parity with chat.

**No streaming in v1.** TrOCR generations are short (~10-50 tokens); Nougat generations can be long but the typical consumer is a batch script, not a live UI. Adding SSE later is reverse-compatible (a `stream: true` form field).

## Runtime

Single class: **`HFVision2SeqRuntime`** in `src/muse/modalities/image_ocr/runtimes/hf_vision2seq.py`.

Wraps:
- `transformers.AutoModelForVision2Seq` for the encoder-decoder
- `transformers.AutoProcessor` for the per-family preprocessing (TrOCR uses `TrOCRProcessor`, Nougat uses `NougatProcessor`; `AutoProcessor.from_pretrained` dispatches)

Method:

```python
def ocr(
    self,
    image: PIL.Image,
    *,
    prompt: str | None = None,
    max_new_tokens: int = 512,
    num_beams: int = 1,
) -> OcrResult: ...
```

Generation:
1. `self._processor(image, return_tensors="pt")` produces `pixel_values`.
2. If `prompt` is set: encode it to `decoder_input_ids` and pass into `model.generate(...)`.
3. `outputs = self._model.generate(pixel_values=..., max_new_tokens=..., num_beams=...)`.
4. `text = self._processor.batch_decode(outputs, skip_special_tokens=True)[0]`.
5. Some models (Nougat) ship a `processor.post_process_generation` for LaTeX cleanup; runtime calls it when present.

Lazy imports follow muse's pattern: torch, AutoModelForVision2Seq, AutoProcessor as module-top sentinels populated by `_ensure_deps()`. Tests patch the sentinels directly.

Device + dtype + inference-mode through `muse.core.runtime_helpers` (the meta-test in `tests/core/test_runtime_helpers_meta.py` AST-scans for re-implementations).

## Bundled (1)

**`trocr-base-printed`** in `src/muse/models/trocr_base_printed.py`. Subclass of `HFVision2SeqRuntime` that exposes `MANIFEST` + `Model = HFVision2SeqRuntime` for discovery.

## HF plugin

New file: `src/muse/modalities/image_ocr/hf.py`. Sniffs:
- `image-to-text` tag (the canonical HF tag for vision-encoder-decoder)
- Repo-name fallback for `trocr`, `nougat`, `texteller` substrings (some checkpoints ship without the canonical tag)

Does NOT claim `image-text-to-text` tag (those are VLMs, future #97 modality).

Priority **110**, between file-pattern plugins (100) and the text-classification catch-all (200). Same priority slot as image_segmentation, audio_embedding, video_generation.

## Capability flags (advisory only)

`supports_handwritten`, `supports_math`. NOT enforced server-side; surfaced in `/v1/models` for client-side filtering. Users get whatever the loaded model can do.

If we want hard enforcement later (a request `prompt` containing math triggers a check), that's a follow-up. v1 is permissive.

## Out of scope

- Donut / DocVQA / structured document parsing.
- Streaming SSE.
- Multi-page input (one image per request).
- PDF input (user pre-rasterizes).
- Batch (one image per request; users can fan out client-side).

## Test strategy

- **Protocol + codec**: `OcrResult` dataclass, OpenAI envelope shape, usage block.
- **Runtime**: mocked `AutoModelForVision2Seq` + `AutoProcessor`. Test ocr() returns text, prompt forwarding, max_new_tokens forwarding, lazy-import discipline. Test post_process_generation hook when present.
- **Route**: FakeModel; multipart upload, capability passthrough, OpenAI error envelope on 400/404/500.
- **Bundled script**: manifest keys, capability flags, runtime inheritance.
- **HF plugin**: image-to-text tag dispatches; repo-name fallback works; `image-text-to-text` (VLM) NOT claimed.
- **Slow e2e**: in-process supervisor + bundled trocr + a small image. Skip if HF cache unavailable.
- **Smoke matrix**: bundled `trocr-base-printed` joins.
