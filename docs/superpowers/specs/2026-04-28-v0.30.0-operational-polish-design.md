# v0.30.0 operational polish (#125, #110, #140)

**Date:** 2026-04-28
**Driver:** bundle three independent operational improvements into one
release. Closes tasks #125, #110, #140.

This is **not** a new modality, **not** a new wire contract. It is
operational polish on three pain points the user has hit repeatedly in
production:

1. **#125** -- `muse serve` is unreachable for 30-60 seconds after
   start because the supervisor waits for ALL workers (one per venv
   group) to pass `/health` before the gateway boots. With six-plus
   enabled models that all want to load weights, the gateway is dead
   air for the duration of the slowest worker.
2. **#110** -- some bundled scripts under `src/muse/models/` declare
   `pip_extras` that miss transitive deps the runtime imports at load
   time (e.g., `transformers`, `accelerate`, `Pillow`, `safetensors`).
   Per-model venvs created from `muse pull <id>` end up importing
   modules that don't exist; the worker exits during model
   construction. The user has fixed this by hand at least three
   times.
3. **#140** -- when `muse[server]` itself adds a runtime dep (e.g.,
   `python-multipart` in v0.13.1), existing per-model venvs created
   with the older `muse[server]` don't auto-update. The user has
   manually run `~/.muse/venvs/<id>/bin/pip install -e
   .[server,...] --upgrade` per venv. There is no CLI command to do
   this systematically.

Each change is small, scoped, and orthogonal to the others. The
release bundles them so users get all three with one upgrade.

## Goal

1. Supervisor starts the gateway as soon as the FIRST worker passes
   `/health`. Late-loading workers continue on a background thread
   that promotes them to `state.workers` once ready, so they
   contribute to `/v1/models` progressively.
2. All 14 bundled scripts under `src/muse/models/` declare a complete
   `pip_extras` tuple. A regression-guard test asserts (via static
   import inspection of each script's runtime path) that the declared
   set covers what the runtime actually imports.
3. `muse models refresh` re-installs `muse[server]` plus the model's
   modality extras into per-model venvs. Supports single-id, `--all`,
   `--enabled`, `--no-extras`, `--json`. Failure-resilient: one venv
   failing does not abort the rest.

## Non-goals

- **Concurrent worker spawn.** Workers still spawn sequentially in
  `run_supervisor`'s loop. The change is on the WAIT side: we wait
  for the first only, then continue waiting for the rest in the
  background.
- **Hot-reload of pip_extras.** `muse models refresh` operates on
  inactive venvs (or the live ones, but does NOT restart the
  supervisor). Picking up a refreshed venv requires `muse serve`
  restart. This is documented; the operator can chain `pkill -f 'muse
  _worker' && muse serve` if they want a hard refresh.
- **Auto-detect transitive deps at install time.** `pip_extras` stays
  manually curated. The audit's regression-guard test catches missing
  deps via static analysis (look at what the runtime module imports
  vs. what `pip_extras` lists), not via runtime install + import.
  Static analysis is sufficient because every bundled runtime is one
  of a small set of patterns (transformers AutoModel, diffusers
  Pipeline, sentence-transformers, etc.).
- **Per-modality refresh batching.** `muse models refresh --modality
  audio/speech` is filed for v0.next. v0.30.0 ships the three flags
  the user actually needs (`--all`, `--enabled`, single-id).
- **Refresh of the shared muse install itself.** This command refreshes
  the *per-venv* installs of muse[server,extras]; the host-level muse
  install (which runs the supervisor) is the user's pip's job. No
  bootstrap loop.
- **CI smoke-test of fresh per-venv installs.** Tracked as a separate
  task (#124). v0.30.0 ships static auditing only; runtime smoke
  install is filed for the next release.

## Architecture

```
=========================================================================
#125: gateway-start latency
=========================================================================
Today (waits for ALL workers serially):

    spawn(w1)  spawn(w2)  spawn(w3)
        |          |          |
        v          v          v
    wait_ready(w1)  ...  wait_ready(w3)        <- all serial; 30-60s
                                                  before gateway boots
        |
        v
    start_gateway

After (gateway boots after FIRST):

    spawn(w1)  spawn(w2)  spawn(w3)
        |
        v
    wait_ready(w1)                              <- ~5-10s typical
        |
        +--> start_gateway (clients can hit ready models)
        |
        +--> background thread:
                wait_ready(w2)  <- promotes to state.workers when ready
                wait_ready(w3)  <- promotes to state.workers when ready

Gateway aggregations (/v1/models, /health) iterate over a snapshot of
state.workers protected by state.lock. Workers in 'pending' status are
filtered out so /v1/models doesn't surface IDs that aren't yet
serving.

=========================================================================
#110: pip_extras audit
=========================================================================

For each bundled script in src/muse/models/<id>.py:

    1. Walk the runtime path: the Model class's __init__ deferred
       imports plus any module-level imports the runtime touches at
       load time.
    2. Compare against the declared pip_extras tuple.
    3. Flag missing deps; add to the manifest with a one-line comment
       explaining why.

A new test (tests/models/test_pip_extras_audit.py) statically inspects
each bundled script and asserts every imported PyPI module appears in
pip_extras. Maps Python module name -> PyPI dist name via a small
table (e.g., 'PIL' -> 'Pillow', 'sentence_transformers' ->
'sentence-transformers'). The test fails on regression.

=========================================================================
#140: muse models refresh
=========================================================================

           muse models refresh <id>
                |
                v
    +-------+------------+
    |  resolve venv path | reads ~/.muse/catalog.json
    +-------+------------+
            |
            v
    +-------+------------+
    | pick muse extras   | infer from manifest.modality:
    |                    |   audio/speech    -> [audio]
    |                    |   image/generation-> [images]
    |                    |   embedding/text  -> [embeddings]
    |                    |   etc.
    +-------+------------+
            |
            v
    +-------+------------------+
    |  pip install -e <muse>   |
    |    [server,<extras>...]  |  always include 'server'
    |    --upgrade             |  always include the model's pip_extras
    +-------+------------------+
            |
            v
    report success | failure (continue past failures with --all)
```

## #125: supervisor boot ordering

### Current behavior

`run_supervisor` in `src/muse/cli_impl/supervisor.py`:

```python
specs = plan_workers()
state = SupervisorState(workers=specs, device=device)
set_supervisor_state(state)

for spec in specs:
    spawn_worker(spec, device=device)        # all spawned

for spec in specs:
    wait_for_ready(port=spec.port)            # all serially blocked
    spec.status = "running"

# THEN gateway starts
routes = [...]
app = build_gateway(routes)
uvicorn.run(app, ...)
```

The user-visible problem: `muse serve` reports "starting gateway on
0.0.0.0:8000" only after every spawned worker passes `/health`.
Six-plus models loading concurrently means the gateway is dead air for
the slowest worker's load time, easily 30-60 seconds.

### New behavior

```python
specs = plan_workers()
state = SupervisorState(workers=specs, device=device)
set_supervisor_state(state)

for spec in specs:
    spawn_worker(spec, device=device)

# Wait for FIRST worker only. With one model the loop terminates on
# the first iteration; with N models it returns as soon as one
# becomes ready. _wait_for_first_ready also tracks failures: if every
# spawned worker fails to come up before deadline, raise.
first_ready = _wait_for_first_ready(specs, timeout=60.0)
first_ready.status = "running"

# Promote remaining workers asynchronously. The background thread
# updates state.workers in place + records status under state.lock so
# gateway aggregations stay consistent.
remaining = [s for s in specs if s is not first_ready]
boot_thread = threading.Thread(
    target=_promote_workers,
    args=(remaining, state),
    daemon=True,
    name="muse-late-boot",
)
boot_thread.start()

# Auto-restart monitor as today.
monitor_thread = threading.Thread(target=_monitor_workers, ...)
monitor_thread.start()

# Build gateway routes from CURRENT state.workers (only the first is
# running; the rest are 'pending' and filtered by gateway routes).
routes = build_routes_from_state(state)
app = build_gateway(routes, state=state)
uvicorn.run(app, ...)
```

The gateway is reachable as soon as the first worker is healthy. Late
workers join the routing table when they pass `/health`; clients that
target their model id get a clear "model not yet ready" error
(503 model_loading) until the worker promotes to running. Once
running, the route fills in.

### State.workers thread safety

`SupervisorState.workers` is mutated by:
- the boot thread (status field flips pending -> running)
- the monitor thread (failure_count, restart_count, status)
- the admin path (enable/disable adds/removes WorkerSpec entries)

Reads happen from:
- the gateway's `/v1/models` and `/health` aggregations
- the admin's `/v1/admin/workers` endpoint
- the boot/monitor threads themselves

`SupervisorState.lock` (an `RLock`, already present in v0.28.0) gates
both reads and writes. Aggregations take the lock briefly to snapshot
the worker list, then release before issuing httpx calls. This keeps
contention low while preserving consistency.

### Gateway: handle pending workers

`build_gateway` currently builds its `app.state.routes` dict
once at startup from a fixed `routes: list[WorkerRoute]`. With
late-promoting workers, the routing table needs to be live.

Two options:
1. Make `app.state.routes` a function that re-derives routes from
   `state.workers` on every request.
2. Have the boot thread call `app.state.routes.update(...)` once a
   worker promotes.

We pick **(1)** for simplicity. The gateway holds a reference to
`SupervisorState` and re-derives routes per request:

```python
def _routes_for(state):
    out = {}
    with state.lock:
        for spec in state.workers:
            if spec.status != "running":
                continue
            url = f"http://127.0.0.1:{spec.port}"
            for m in spec.models:
                out[m] = WorkerRoute(model_id=m, worker_url=url)
    return out
```

Cost is one dict-build per request. With ~10 workers that's a few
microseconds; acceptable. The code stays simple and the boot thread
doesn't have to mutate gateway state.

For admin endpoints that ALSO need this, the helper lives in
`muse.cli_impl.gateway` and `muse.admin.routes` calls it.

### `_wait_for_first_ready` semantics

```python
def _wait_for_first_ready(
    specs: list[WorkerSpec],
    *,
    timeout: float = 60.0,
    poll_interval: float = 0.5,
) -> WorkerSpec:
    """Block until ANY worker passes /health.

    Returns the first one that becomes ready. Raises TimeoutError if
    no worker passes within timeout (i.e. every spawn failed).

    Pure round-robin polling: in each tick poll all specs once; a
    worker that's already running is returned immediately.
    """
```

This ensures we don't deadlock if the spec.port that happens to come
up first is buried behind a slower worker. Round-robin is simple and
correct.

### `_promote_workers` semantics

```python
def _promote_workers(
    specs: list[WorkerSpec],
    state: SupervisorState,
    *,
    timeout: float = 120.0,
) -> None:
    """Background: wait for each remaining worker; promote on /health.

    Runs in a daemon thread. For each spec:
      - polls /health with the existing wait_for_ready helper;
      - on success, sets spec.status = "running" under state.lock;
      - on timeout, sets spec.status = "unhealthy" (the monitor's
        next tick will trigger a restart attempt).
    """
```

Failures here are non-fatal. The auto-restart monitor (already in
place) handles the resurrection logic; this thread just promotes
fresh workers from pending to running.

### Regression-guard test

`tests/models/test_pip_extras_audit.py`:

```python
"""Static check: every bundled script declares pip_extras covering
its runtime imports.

Maps Python module names to PyPI dist names via a small table; flags
imports of modules whose dist isn't in pip_extras. Defends against
the fresh-venv install path that has bitten production multiple
times.
"""
```

The test:
1. For each `src/muse/models/*.py` other than `__init__.py`:
   a. Static-walk imports (top-level + inside-function).
   b. Filter to "real PyPI deps" via a mapping table.
   c. Read MANIFEST['pip_extras'] from the module.
   d. Assert every PyPI dist found in (b) appears in (c).

PyPI dist mapping (extensible):

```python
MODULE_TO_PYPI = {
    "torch": "torch",
    "transformers": "transformers",
    "diffusers": "diffusers",
    "accelerate": "accelerate",
    "PIL": "Pillow",
    "safetensors": "safetensors",
    "sentence_transformers": "sentence-transformers",
    "scipy": "scipy",
    "inflect": "inflect",
    "unidecode": "unidecode",
    "numpy": "numpy",
    "librosa": "librosa",
    "kokoro": "kokoro",
    "misaki": "misaki",
    "soundfile": "soundfile",
    "einops": "einops",
    "imageio": "imageio",
}
```

Modules NOT in this table (`muse`, `os`, `logging`, etc.) are skipped
as part of the standard library or this project itself. New bundled
scripts that import a new PyPI module raise a clear error in the
test:

```

is not in MODULE_TO_PYPI; add the mapping to enable the audit.
```

This makes the audit self-extending: the maintainer either adds the
PyPI dist (so the audit can verify it's in pip_extras) or maps it as
stdlib/local (so it's skipped).

## #140: muse models refresh

### CLI surface

```
muse models refresh <model-id>           # one venv
muse models refresh --all                # every pulled venv
muse models refresh --enabled            # only enabled venvs
muse models refresh <id> --no-extras     # only refresh muse[server], skip pip_extras
muse models refresh --all --json         # machine-readable progress
```

### Behavior

For each target venv:

```
<venv>/bin/pip install --upgrade -e <muse-repo-path>[server,<modality-extra>]
```

then, unless `--no-extras`:

```
<venv>/bin/pip install --upgrade <pip_extras...>
```

Returns one of:
- `"ok"` -- pip exit 0
- `"failed"` -- pip exit non-zero (output captured)
- `"skipped"` -- venv missing or stale catalog entry

Continue past failures. Aggregate at the end.

### Modality-to-extras mapping

Inferred from the model's modality. Hand-maintained in
`src/muse/cli_impl/refresh.py`:

```python
MODALITY_EXTRAS = {
    "audio/speech": ["audio"],
    "audio/transcription": [],          # no extra; muse[server] suffices
    "audio/embedding": [],
    "audio/generation": [],
    "image/generation": ["images"],
    "image/animation": ["images"],
    "image/upscale": ["images"],
    "image/embedding": [],
    "image/segmentation": [],
    "embedding/text": ["embeddings"],
    "text/classification": [],
    "text/rerank": [],
    "text/summarization": [],
    "video/generation": ["images"],
    "chat/completion": [],
}
```

For `audio/speech`, additional model-specific extras land via the
manifest's `pip_extras` (kokoro-82m needs `kokoro` etc.); we don't
re-encode that as a muse extra. The MODALITY_EXTRAS map is for the
muse-side extra-name, not the model's PyPI deps.

### Implementation outline

`src/muse/cli_impl/refresh.py`:

```python
"""`muse models refresh`: re-install muse[server,<extras>] into per-model venvs.

Use case: muse[server] gets a new dep (e.g. python-multipart in v0.13.1)
and existing venvs are stale. This command upgrades them in-place.
"""
from __future__ import annotations

import json
import logging
import subprocess
import sys
from dataclasses import dataclass
from pathlib import Path

from muse.core.catalog import _read_catalog, get_manifest, is_enabled

logger = logging.getLogger(__name__)


# Map a modality tag to the muse-side optional-deps extras it needs.
# `server` is added unconditionally.
MODALITY_EXTRAS: dict[str, list[str]] = {
    "audio/speech": ["audio"],
    "audio/transcription": [],
    ...
}


@dataclass
class RefreshResult:
    model_id: str
    state: str            # "ok" | "failed" | "skipped"
    message: str = ""
    pip_output: str = ""


def _infer_extras(modality: str) -> list[str]:
    return MODALITY_EXTRAS.get(modality, [])


def _muse_repo_root() -> Path:
    """Locate the muse source tree to install in editable mode.

    Falls back to PyPI install if we're running from a wheel (no
    pyproject.toml in cwd's ancestors).
    """
    here = Path(__file__).resolve()
    for parent in here.parents:
        if (parent / "pyproject.toml").exists():
            return parent
    return Path.cwd()


def refresh_one(
    model_id: str,
    *,
    no_extras: bool = False,
) -> RefreshResult:
    """Refresh a single venv. Returns a RefreshResult."""
    catalog = _read_catalog()
    entry = catalog.get(model_id)
    if not entry:
        return RefreshResult(model_id, "skipped", "not in catalog")
    python_path = entry.get("python_path")
    if not python_path or not Path(python_path).exists():
        return RefreshResult(
            model_id, "skipped",
            f"python_path missing or not found: {python_path}",
        )
    manifest = get_manifest(model_id) or {}
    modality = manifest.get("modality") or entry.get("modality") or ""
    pip_extras = list(manifest.get("pip_extras") or ())
    muse_extras = _infer_extras(modality)
    spec = "[server" + ("," + ",".join(muse_extras) if muse_extras else "") + "]"
    target = str(_muse_repo_root()) + spec

    cmd = [python_path, "-m", "pip", "install", "--upgrade", "-e", target]
    logger.info("refresh %s: %s", model_id, " ".join(cmd))
    proc = subprocess.run(cmd, capture_output=True, text=True)
    if proc.returncode != 0:
        return RefreshResult(
            model_id, "failed",
            "muse[server] install failed",
            proc.stdout + proc.stderr,
        )

    if not no_extras and pip_extras:
        cmd2 = [python_path, "-m", "pip", "install", "--upgrade", *pip_extras]
        logger.info("refresh %s extras: %s", model_id, " ".join(cmd2))
        proc2 = subprocess.run(cmd2, capture_output=True, text=True)
        if proc2.returncode != 0:
            return RefreshResult(
                model_id, "failed",
                "pip_extras install failed",
                proc2.stdout + proc2.stderr,
            )

    return RefreshResult(model_id, "ok")


def run_refresh(
    *,
    model_id: str | None = None,
    all_: bool = False,
    enabled_only: bool = False,
    no_extras: bool = False,
    as_json: bool = False,
) -> int:
    """Entry point for `muse models refresh`."""
    catalog = _read_catalog()
    targets: list[str]
    if all_:
        targets = sorted(catalog.keys())
    elif enabled_only:
        targets = sorted(mid for mid in catalog if is_enabled(mid))
    elif model_id is not None:
        targets = [model_id]
    else:
        print("error: pass <model_id>, --all, or --enabled", file=sys.stderr)
        return 2

    if not targets:
        print("no targets selected")
        return 0

    results = [refresh_one(mid, no_extras=no_extras) for mid in targets]

    if as_json:
        print(json.dumps([r.__dict__ for r in results], indent=2))
    else:
        for r in results:
            mark = {"ok": "OK", "failed": "FAIL", "skipped": "skip"}[r.state]
            line = f"  {r.model_id:24s} [{mark:5s}] {r.message}"
            print(line)

    n_ok = sum(1 for r in results if r.state == "ok")
    n_failed = sum(1 for r in results if r.state == "failed")
    n_skipped = sum(1 for r in results if r.state == "skipped")
    print(f"\n{n_ok} ok, {n_failed} failed, {n_skipped} skipped")
    return 0 if n_failed == 0 else 1
```

## Resilience considerations

1. **Race in `_promote_workers`**: a worker that promotes to running
   while admin is mid-`disable` could have its slot mutated underneath.
   The `state.lock` already serializes admin ops; the boot thread takes
   the same lock when flipping status.
2. **Gateway sees stale state**: with per-request route building from
   `state.workers`, a request issued the instant before a worker
   promotes still gets the old (no-route) view. The user retries; on
   the next request the route fills in. No data corruption.
3. **`_wait_for_first_ready` deadlock**: if every spawned worker
   crashes before any responds 200, the function would never return.
   Bound by the `timeout` argument; raises `TimeoutError` past
   deadline. `run_supervisor` propagates the error so the caller
   knows the supervisor never came up.
4. **pip_extras audit false positives**: a runtime that uses
   `importlib` to load a module won't be detected by static analysis.
   Mitigation: every bundled runtime in muse uses plain
   `import`/`from ... import ...`; if a future runtime needs runtime
   imports, the audit is a "best effort" check, not a hard guarantee.
   This is documented in the test docstring.
5. **`muse models refresh --all` partial failure**: documented; the
   failure summary tells the user which venvs failed and prints their
   pip output for diagnosis. Exit code 1 signals to scripts that not
   every refresh succeeded.
6. **`_muse_repo_root` heuristic**: walks parents looking for
   `pyproject.toml`. If muse is installed from a wheel, this falls
   back to cwd. Documented; users who installed from PyPI can still
   run `muse models refresh` (the venv pip will pick up muse from
   PyPI rather than editable). The command remains useful.

## Out of scope

- Concurrent worker spawn (sequential today; might revisit if startup
  latency is still problematic with 20+ models)
- Hot-reload of pip_extras after refresh (operator restarts manually)
- Auto-detect transitive deps at install time
- Per-modality refresh batching (`--modality audio/speech`)
- Refresh of the host muse install itself
- CI smoke-test of fresh per-venv installs (#124)
- WebSocket progress for long-running refreshes

## Migration

Zero migration cost.

- `#125` is a behavior change in the supervisor; gateway becomes
  available faster. No CLI flag, no opt-in.
- `#110` is manifest-only changes for four scripts (bark_small,
  kokoro_82m, mert_v1_95m, soprano_80m) plus a regression-guard
  test. Existing pulled venvs are unaffected (their pip_extras was
  installed at pull time).
- `#140` is a brand-new CLI subcommand; users who don't run it see
  no change.

The user-facing announce text is one paragraph in the v0.30.0
release notes covering all three.

## Why this matters

- **#125** turns dead-air startup time into useful warm-up time.
  With six models and a 30 s slowest-worker load, the supervisor
  previously kept the gateway dark for 30 s; now it serves five of
  the six immediately.
- **#110** prevents the "I just `muse pull foo`'d and the worker
  exited" failure mode. The audit fixes today's gaps; the
  regression-guard test prevents tomorrow's.
- **#140** turns a multi-step manual workaround into one command.
  When `muse[server]` adds a new dep, `muse models refresh --all`
  upgrades every pulled venv in one pass.

Together: faster startup, fewer fresh-install footguns, and a
single command for the recurring "muse[server] got a new dep" pain.
