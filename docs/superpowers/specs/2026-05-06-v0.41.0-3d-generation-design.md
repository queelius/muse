# v0.41.0: 3D Generation Modality

**Status:** Spec, ready for implementation.
**Author:** Alex Towell
**Date:** 2026-05-06

## Goal (one sentence)

Add a `3d/generation` modality with two routes (`POST /v1/3d/generations` for text-to-3d, `POST /v1/3d/from-image` for image-to-3d), GLB output, TripoSR bundled (image-to-3d), TRELLIS curated (both routes), capability-gated dispatch.

## Motivation

3D generation is one of the few major modality categories muse hasn't shipped yet. Industry-wide, image-to-3d is the dominant 3D generation use case in 2026 (TripoSR, Wonder3D, InstantMesh, TRELLIS); text-to-3d remains a smaller niche but is genuinely useful for prototyping and asset generation. With v0.40.x's lazy load + LRU eviction in place, muse can host 3D models alongside text/image/audio/video without the operational overhead of a separate platform.

## Design summary

| Concept | Choice |
|---|---|
| Output format | GLB only (binary glTF; embedded textures; web-friendly) |
| Wire pattern | Stability.ai-shape sync, blocking. No async/JobStore. |
| MIME tag | `3d/generation` |
| Python pkg | `model_3d_generation/` (digit-prefix Python identifier dodge) |
| Routes | `POST /v1/3d/generations` (text), `POST /v1/3d/from-image` (multipart) |
| Bundled default | TripoSR (image-to-3d only, ~120MB, MIT, CUDA) |
| Curated coverage | TRELLIS (both routes), Hunyuan3D-2, Wonder3D as v1 set |
| Capability gates | `supports_text_to_3d`, `supports_image_to_3d` (400 if mismatch) |

### Wire shape

#### Text-to-3d

```
POST /v1/3d/generations
Content-Type: application/json

{
  "model": "trellis-text",
  "prompt": "a small wooden chair, oak finish",
  "n": 1,                          // optional; capped at 2 (heavy)
  "seed": 42,                      // optional
  "response_format": "b64_json"    // "b64_json" (default) | "url"
}

200 OK
{
  "created": 1746500000,
  "model": "trellis-text",
  "data": [
    {"b64_json": "<base64-encoded GLB>", "format": "glb"}
  ]
}
```

#### Image-to-3d

```
POST /v1/3d/from-image
Content-Type: multipart/form-data

image: <PNG/JPEG file>
model: triposr
n: 1
seed: 42
response_format: b64_json

200 OK
{
  "created": 1746500000,
  "model": "triposr",
  "data": [
    {"b64_json": "<base64-encoded GLB>", "format": "glb"}
  ]
}
```

### Error envelopes

- 400 `unsupported_route`: model declares only `supports_image_to_3d=True` but request hit `/v1/3d/generations` (text route). Message names which capability is missing.
- 400 `invalid_image`: empty upload, malformed file, exceeds size cap.
- 404 `model_not_found`: standard.
- 503 `model_too_large_for_device`: lazy-load eviction couldn't free enough memory.
- 500 `internal_error`: backend exception during generation; the message includes the original exception text.

### Capability flags

Per the v0.20.x convention (each modality declares its routes via boolean caps; the route-handler 400s on mismatch):

```yaml
capabilities:
  memory_gb: 1.5            # peak inference, like other modalities
  device: cuda               # 3D models are virtually always GPU-only
  supports_text_to_3d: false
  supports_image_to_3d: true
  output_format: glb
```

Bundled `triposr`: `supports_image_to_3d: true`, `supports_text_to_3d: false`.
Curated `trellis-text-image`: both true.

A request to `/v1/3d/generations` with a model declaring `supports_text_to_3d=False` returns 400 immediately (gateway-level check; never hits the runtime).

### Image input handling

Image-to-3d's input image is multipart/form-data, mirroring `/v1/images/edits` and `/v1/images/upscale` (v0.21.0 + v0.25.0). Reuse the existing image-input helpers in `muse.modalities.image_generation.routes` (size cap via `MUSE_IMAGE_INPUT_MAX_BYTES`, MIME validation, decode).

A separate cap `MUSE_3D_INPUT_MAX_BYTES` (default 20 MiB) handles 3D-specific tuning if needed; default the env-tunable cap to the same value as image but allow operators to bump it.

### Codec

GLB is binary glTF. The runtime returns a `bytes` blob; the codec encodes as base64 for `b64_json` mode, or writes to a temp file and returns a path-based URL for `url` mode. The protocol's `Generation3DResult` dataclass holds `glb_bytes: bytes` plus metadata.

`format` field on each data element is always `"glb"` for v0.41.0; reserved as a string so future formats (USDZ, OBJ-archive) are additive.

### Bundled: TripoSR

- HF repo: `stabilityai/TripoSR`
- License: MIT
- Size: ~120MB safetensors
- Memory: ~1.5GB peak on CUDA at default 256x256 input resolution
- Routes claimed: `image-to-3d` only (`supports_image_to_3d: true`, `supports_text_to_3d: false`)
- Runtime class: `TripoSRRuntime` in `runtimes/triposr.py`. Uses `tsr.system.TSR.from_pretrained` (TripoSR's own loader; not a transformers AutoModel) plus the `trimesh` library to convert TripoSR's mesh output into a GLB blob.
- pip_extras: `torch>=2.1.0`, `torchvision>=0.16.0`, `transformers>=4.40.0`, `trimesh>=4.0`, `tsr` (TripoSR's PyPI package), `Pillow`, `numpy`, `omegaconf`

### Curated (v0.41.0 set)

| id | URI | Routes | Notes |
|---|---|---|---|
| `trellis-text-image` | `hf://JeffreyXiang/TRELLIS-image-large` (also has text path) | both | Uses transformers + custom code (`trust_remote_code=True`); ~1.5GB at fp16 |
| `wonder3d-base` | `hf://flamehaze1115/wonder3d-v1.0` | image | Multi-view diffusion for 3D, ~3GB |

Curated runtime path can be `transformers`-based (TRELLIS, Hunyuan3D-2) or `diffusers`-based (Wonder3D, Shap-E). The HF plugin sniffs by repo-name allowlist + tag-based fallback, returning `runtime_path` to the right runtime class.

### Probe defaults

`PROBE_DEFAULTS = {"shape": "1 small synthetic image", "call": _probe_call}` where `_probe_call` builds a 256x256 RGB PNG, writes to a temp file, calls `backend.image_to_3d(path)`. Mirrors audio_classification's probe-with-temp-file pattern.

### Memory accounting

GLB output blobs are typically 1-50MB. The codec's runtime memory for the b64_json encode is ~2x the blob size (the original + the encoded string). At 50MB GLB this is 100MB peak in process. Acceptable; no per-call memory cap beyond the existing image input cap.

The model itself (TripoSR ~1.5GB, TRELLIS ~3GB at fp16) follows the standard lazy-load + LRU pattern; nothing special.

### Lock discipline

Per-backend `_inference_lock` (the v0.34.0 pattern) serializes calls into a single 3D model. A 3D inference call is ~10-30s; without the lock, two concurrent requests on the same model would either fight for VRAM or step on each other's pipeline state.

## What stays the same

- Catalog format (additive: new modality tag).
- HTTP wire compat for unrelated routes.
- Lazy-load + LRU eviction (v0.40.x).
- Idle eviction (v0.40.1).
- All existing CLI verbs (`muse pull`, `muse models warmup`, etc.).

## What changes for users

- New modality `3d/generation` appears in `muse models list`, `/v1/models`, `muse search --modality 3d/generation`.
- New routes `POST /v1/3d/generations` and `POST /v1/3d/from-image` accept `model` field referencing any pulled 3D model.
- `muse pull triposr` installs the bundled image-to-3d default.
- `muse pull trellis-text-image` (or any other curated) installs that.

## Out of scope (deferred)

- **USDZ output** (Apple AR format): a stretch goal. GLB covers ~95% of web 3D use; USDZ matters mostly for native iOS AR. Add if there's demand.
- **OBJ + MTL archive output**: legacy desktop 3D pipeline. Skip; users can convert GLB → OBJ client-side via `gltfpack` or `pymeshlab`.
- **Async / job-based pattern**: the sync pattern matches muse's video/generation precedent; can revisit if 3D models routinely run >120s.
- **Streaming progress** (SSE on long generation): nice-to-have but not necessary; first-class clients (`Generation3DClient`) just wait.
- **Mesh post-processing** (decimation, UV-unwrap, baking): out of muse's scope. Users compose with `pymeshlab`, `Blender`, etc.
- **Per-vertex texture upscaling** as a separate route: future spec if anyone needs it.

## Risks

- **TripoSR's `tsr` PyPI package quality**: it's the official TripoSR package but is sparsely maintained. If it breaks on a future torch / transformers, we mitigate with a pinned version range in pip_extras.
- **TRELLIS requires `trust_remote_code=True`**: transformers warns about arbitrary code execution. Documented in CLAUDE.md per the existing convention; the curated yaml includes a `trust_remote_code: true` capability flag the runtime forwards.
- **GLB blob size variance**: a 256x256 input → ~5MB GLB. A high-resolution single-image-to-3d at higher quality settings can produce ~50MB GLB. Either fits in memory.

## Testing strategy

### Unit tests

- `tests/modalities/model_3d_generation/test_codec.py`: GLB-bytes-to-b64_json encode; URL mode writes a tmp file; envelope shape matches `data[0].format == "glb"`.
- `tests/modalities/model_3d_generation/test_protocol.py`: `Generation3DResult` shape, `Generation3DBackend` protocol structural typing.
- `tests/modalities/model_3d_generation/test_routes.py`: text route 200 with valid prompt; image route 200 with multipart; capability gating returns 400 for wrong route; 404 unknown model; 400 empty image; 500 backend exception.
- `tests/modalities/model_3d_generation/test_client.py`: `Generation3DClient.from_text(prompt, model=...)` and `from_image(image, model=...)` smoke tests with patched httpx.
- `tests/modalities/model_3d_generation/test_hf_plugin.py`: sniff true on `image-to-3d` tag + repo-name allowlist; sniff true on TRELLIS-shaped repos; sniff false on random.
- `tests/modalities/model_3d_generation/runtimes/test_triposr.py`: mocked-deps tests for the runtime (skip-on-no-deps, deferred-import sentinel pattern).
- `tests/models/test_triposr.py`: bundled-script smoke test.

### Slow e2e

- `@pytest.mark.slow` test that loads the real TripoSR model (CUDA required), runs `image_to_3d` on a synthetic image, asserts the GLB blob is non-empty and round-trips through trimesh.
