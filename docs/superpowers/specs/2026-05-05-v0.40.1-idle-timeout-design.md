# v0.40.1: Per-Model Idle Timeout

**Status:** Spec, ready for implementation.
**Author:** Alex Towell
**Date:** 2026-05-05

## Goal (one sentence)

Add a background sweeper that unloads loaded models which have been idle (no requests) for longer than the model's declared `idle_timeout_seconds`, freeing memory without waiting for traffic-driven LRU eviction.

## Motivation

v0.40.0 ships memory-pressure eviction: a request that doesn't fit triggers LRU eviction. That works, but in practice operators want their long-idle models GC'd proactively so the fleet doesn't sit on stale VRAM/RAM:

- A user fires `muse pull qwen3.5-9b-q4` for one Saturday afternoon experiment, then doesn't use it for three days. With v0.40.0 alone, that 9GB sits in VRAM until *something else* triggers eviction.
- Operators on shared boxes want to release memory back to the OS for other workloads when muse isn't actively serving a model.
- Avoids the "memory pressure cliff" , instead of waiting for a request to fail eviction, the system stays slack and predictable.

## Design summary

| Concept | v0.40.0 | v0.40.1 (this spec) |
|---|---|---|
| Eviction trigger | Memory pressure (LRU) | Memory pressure (LRU) OR idle timeout |
| Idle eviction | None | Optional per-model `idle_timeout_seconds`; default `null` (never idle-evict) |
| Sweeper | None | Background thread; default 30s interval; tunable via env |
| Default behavior | Models stay loaded forever once warm | Same (idle_timeout defaults null = preserve v0.40.0 semantics) |

### Mechanics

**Manifest field (optional):**

```yaml
capabilities:
  memory_gb: 4.0
  device: cuda
  idle_timeout_seconds: 1800   # null or absent = never idle-evict
```

`idle_timeout_seconds: null` (or absent) is the default. Operators who want idle eviction declare it per-model.

**Sweeper:**

A background thread on the supervisor (`LoadDirector.idle_sweeper` or a sibling thread, TBD by implementer). Runs every `MUSE_IDLE_SWEEP_INTERVAL_SECONDS` (default 30). On each tick:

1. Snapshot `director.loaded` under the director lock (existing `status()` returns enough; otherwise read directly).
2. For each `LoadEntry` with `refcount == 0`, look up the model's `idle_timeout_seconds` from its catalog manifest.
3. If `idle_timeout_seconds is None`: skip (never idle-evict).
4. If `time.monotonic() - last_touched_at >= idle_timeout_seconds`: idle eviction candidate.
5. For each candidate, call `disable_fn(model_id)` (same primitive as memory-pressure eviction). Refcount-protect: if a request lands between snapshot and disable_fn (refcount > 0 by the time we evict), skip , re-check under lock.
6. Append a `DecisionLogEntry(action="evict", reason=f"idle_timeout:{seconds}s", evicted=[model_id])` to `recent_decisions`.

**Lock discipline:**

- Snapshot under `director.lock`. Drop lock.
- Refcount re-check + LoadEntry pop is again under `director.lock` (just before calling `disable_fn`, like the memory-pressure path).
- `disable_fn` runs OUTSIDE the lock (slow I/O).
- DecisionLogEntry append under lock.

**Auto-restart monitor compatibility:**

The existing monitor polls `state.workers` for crashed workers. The idle sweeper drives `state.workers` via `disable_fn` (which goes through `unload_model_from_worker`, the v0.40.0 op that mutates state.workers in-place). No new race surface; the monitor sees workers come and go uniformly regardless of trigger.

**Configuration:**

```bash
MUSE_IDLE_SWEEP_INTERVAL_SECONDS=30   # default; tunable
```

No "disable idle eviction globally" env var: just don't set `idle_timeout_seconds` on any manifest. The sweeper still ticks but finds nothing to do (~free).

**Shutdown:**

The sweeper thread is daemon; it exits when the supervisor process does. On graceful supervisor shutdown (SIGTERM), the sweeper joins via the existing `state.stop_event` already used by the auto-restart monitor.

### What stays the same

- LoadDirector's three-phase acquire / release / on-demand LRU eviction (v0.40.0).
- Catalog format (idle_timeout_seconds is purely additive).
- HTTP wire compat.
- Admin REST endpoints.
- The five-state status enum: idle eviction transitions a model from `enabled_loaded` to `enabled_unloaded`, no new states.

### What changes

- Manifest schema: optional `capabilities.idle_timeout_seconds` (positive number; null/absent → no idle eviction).
- New supervisor thread: idle sweeper. Configurable interval via `MUSE_IDLE_SWEEP_INTERVAL_SECONDS`.
- `recent_decisions` now distinguishes `reason="idle_timeout:Ns"` from `reason="evicted_for_<other>"`.
- `muse models info` displays the idle_timeout if set.

## Components

### `IdleSweeper` (new)

Lives in `src/muse/cli_impl/idle_sweeper.py` or as a method on `LoadDirector`. Implementer chooses based on coupling. Constructor injection:

```python
class IdleSweeper:
    def __init__(
        self,
        director: LoadDirector,
        catalog_lookup: Callable[[str], dict],  # model_id -> manifest dict
        interval_seconds: float = 30.0,
        stop_event: threading.Event | None = None,
    ): ...

    def start(self) -> threading.Thread: ...
    def tick(self) -> list[str]: ...  # for testability; returns evicted ids
```

`tick()` is the per-iteration body, exposed for unit tests so we don't need to drive the actual thread loop. The thread loop is just `while not stop_event.is_set(): self.tick(); stop_event.wait(self.interval_seconds)`.

### Wiring

`run_supervisor` instantiates `IdleSweeper(director=state.director, catalog_lookup=get_manifest, interval_seconds=...)` and starts it alongside the auto-restart monitor. The sweeper's stop_event shares with the supervisor's existing `state.stop_event`.

## Data flow

### Idle tick (no eviction)

```
sweeper.tick():
  1. with director.lock:
       loaded_snapshot = list(director.loaded.values())  # copy refs
  2. for entry in loaded_snapshot:
       manifest = catalog_lookup(entry.model_id)
       idle_timeout = manifest.get("capabilities", {}).get("idle_timeout_seconds")
       if idle_timeout is None or idle_timeout <= 0:
           continue
       if entry.refcount > 0:
           continue
       if time.monotonic() - entry.last_touched_at < idle_timeout:
           continue
       candidates.append(entry)
  3. for candidate in candidates:
       with director.lock:
         # Re-check: another request may have grabbed the model between
         # the snapshot and now. Refcount > 0 -> skip; entry gone -> skip.
         current = director.loaded.get(candidate.model_id)
         if current is None or current.refcount > 0:
             continue
         director.loaded.pop(candidate.model_id, None)
       try:
           director.disable_fn(candidate.model_id)
       except Exception as e:
           logger.warning("idle sweeper: disable_fn(%s) raised: %s", candidate.model_id, e)
           # Re-insert to avoid orphan worker (same pattern as on-demand
           # eviction, see Task C of v0.40.0)
           with director.lock:
               director.loaded[candidate.model_id] = candidate
           continue
       with director.lock:
         director.recent_decisions.append(DecisionLogEntry(
           timestamp=time.time(),
           model_id=candidate.model_id,
           action="evict",
           memory_gb=candidate.memory_gb,
           free_before_gb=...,
           free_after_gb=...,
           reason=f"idle_timeout:{idle_timeout}s",
           evicted=[candidate.model_id],
         ))
```

### Concurrency with on-demand acquire

If a request arrives mid-sweep:

- Before snapshot: model gets refcount > 0 → skipped on this tick (will be considered next tick).
- During snapshot: snapshot is a list of references; refcount on the LIVE entry may have changed by the time we re-check under lock. Re-check covers this.
- Between re-check and disable_fn: this is where the orphan-worker case from Task C applies. The lock is dropped between `pop` and `disable_fn(...)`, so a fresh acquire calling `enable_fn` could land. The pop ensures the new acquire sees a clean slate (cold load); the now-popped entry's worker eventually dies via disable_fn. This works because the model was IDLE , by definition no traffic, so no fresh acquire is realistically racing.

If `disable_fn` fails: re-insert via the same Task C pattern (retain the LoadEntry's `last_touched_at` so it's NOT immediately re-evicted next tick , it goes to LRU position naturally).

## Error handling

- **`disable_fn` raises:** re-insert the LoadEntry, log warning, continue to next candidate. Same semantics as on-demand eviction's failed-victim handling.
- **Catalog lookup raises (`KeyError` for an evicted/removed model):** skip silently. The model isn't in the catalog anymore; let the existing LoadDirector cleanup handle it on the next acquire (which will also fail).
- **Sweeper thread crashes:** catch all exceptions in the outer loop; log; continue ticking. Don't let one bug take down the sweeper.

## Configuration

```bash
MUSE_IDLE_SWEEP_INTERVAL_SECONDS=30  # default; how often to check
```

Per-model `idle_timeout_seconds` in the manifest. No global default , opt-in only.

## Testing strategy

### Unit tests (no real workers)

`tests/cli_impl/test_idle_sweeper.py`:
- Sweeper picks up an idle candidate (refcount=0, last_touched > timeout).
- Sweeper SKIPs models with `idle_timeout_seconds is None`.
- Sweeper SKIPs models with refcount > 0 (in-flight request).
- Sweeper SKIPs models recently touched (last_touched < timeout).
- `disable_fn` failure: re-inserts LoadEntry, doesn't crash sweeper, processes other candidates.
- DecisionLogEntry recorded with `reason="idle_timeout:Ns"`.
- Catalog lookup raises (model removed) → silently skipped.

### Integration tests (in-process supervisor)

`tests/cli_impl/test_e2e_idle_eviction.py` (slow):
- Boot supervisor with a model that has `idle_timeout_seconds=1`.
- Cold-load the model via a request. Verify it's in director.loaded.
- Wait 1.5s. Run a manual sweeper tick. Verify the model is now unloaded.
- Verify `recent_decisions` has the idle-timeout entry.

### Regression: no idle timeout = no behavior change

`tests/cli_impl/test_idle_sweeper.py::test_no_idle_timeout_means_no_eviction`:
- Sweeper tick over a model with no `idle_timeout_seconds` does nothing, even if `last_touched_at` is hours ago.

## Migration

- Catalog edits are NOT required. Existing manifests without `idle_timeout_seconds` get the v0.40.0 behavior (no idle eviction).
- A bundled model that wants idle eviction declares `capabilities.idle_timeout_seconds: 1800` (or whatever) in its MANIFEST dict. No CLI tools needed to opt in.

## Out of scope (deferred)

- **Global default for `idle_timeout_seconds`** (e.g., `MUSE_DEFAULT_IDLE_TIMEOUT_SECONDS`). Operators can set it per-model; an opinionated global default risks surprising legacy users.
- **Idle eviction telemetry beyond `recent_decisions`.** The existing 20-entry deque is sufficient. A long-form audit log can come later if needed.
- **Adaptive timeout** (e.g., based on observed traffic patterns). Pure timer-driven for v0.40.1.

## Risks

- **Idle eviction kicks in too aggressively for users who didn't realize a manifest declared it.** Mitigation: `muse models info` displays the idle_timeout; it's also visible in the catalog.json.
- **Sweeper interferes with hot acquires** (interleaved disable_fn and acquire on the same model). Mitigation: refcount re-check under lock; re-insert on disable_fn failure; sweeper opportunistic, never required.
- **Sweeper thread leak on supervisor crash.** Mitigation: daemon=True so process exit cuts it; stop_event on graceful shutdown.
