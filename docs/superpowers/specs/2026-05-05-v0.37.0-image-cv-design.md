# v0.37.0 image/cv Design

**Date:** 2026-05-05
**Status:** Approved for plan
**Closes:** #109 image CV primitives (depth, keypoint, object-detection)

## Goal

A new `image/cv` modality with three routes (`POST /v1/images/depth`, `POST /v1/images/keypoints`, `POST /v1/images/detect`), each backed by its own runtime, sharing the modality tag and registry namespace. Three bundled models (one per primitive). Curated alternatives for size and architecture variants.

## Why one modality, not three

The three primitives diverge sharply in *output* shape (depth = pixel map; keypoint/detection = structured records), but converge on *input* (multipart image upload) and *infrastructure* (transformers AutoModel + AutoImageProcessor, shared device/dtype helpers, shared multipart decode). They also tend to be deployed together in real CV pipelines (e.g., detect objects + extract pose for each + estimate scene depth). Bundling them under `image/cv` mirrors how `text/classification` hosts both `/v1/moderations` and `/v1/text/classifications`: one modality tag, multiple routes, capability flags decide which models each route accepts.

The alternative (three separate modalities) would triple the discovery surface, plugin count, and boilerplate without proportionate benefit. Their runtimes don't share inference code, but they share everything around it.

## Wire shapes

### `POST /v1/images/depth`

Multipart input:

| field | type | required | notes |
|---|---|---|---|
| `image` | file | yes | PNG/JPEG/WebP. Subject to `MUSE_IMAGE_INPUT_MAX_BYTES`. |
| `model` | str | no | Catalog id; modality default applies. |
| `response_format` | str | no | `"png16"` (default) or `"float32"`. |

Response:

```json
{
  "id": "depth-<uuid>",
  "model": "depth-anything-v2-small",
  "depth_map": "<base64>",
  "format": "png16",
  "width": 640,
  "height": 480,
  "min_depth": 0.0,
  "max_depth": 1.0,
  "metric_depth": false
}
```

`depth_map` encoding:

- **`png16`**: 16-bit grayscale PNG. Map's float values normalized into [0, 65535] via the response's `min_depth` / `max_depth` bounds. Lossy at 1/65535 of the range; viewable in any image tool.
- **`float32`**: raw `(H * W * 4)` bytes of little-endian float32, base64-encoded. Client reshapes via `width` and `height`. Fully precise.

`metric_depth: true` (ZoeDepth) means values are meters from camera. `metric_depth: false` (Depth-Anything) means relative inverse depth, normalized.

### `POST /v1/images/keypoints`

Multipart input:

| field | type | required | notes |
|---|---|---|---|
| `image` | file | yes | as above |
| `model` | str | no | as above |
| `threshold` | float | no | Per-keypoint visibility threshold, default 0.3. |

Response:

```json
{
  "id": "kp-<uuid>",
  "model": "vitpose-base-simple",
  "image_size": [640, 480],
  "detections": [
    {
      "bbox": [120.5, 80.0, 200.0, 380.0],
      "score": 0.95,
      "keypoints": [
        {"name": "nose", "x": 320.0, "y": 240.0, "score": 0.99},
        {"name": "left_eye", "x": 305.0, "y": 230.0, "score": 0.98}
      ]
    }
  ]
}
```

`image_size` is `[W, H]` (PIL convention, matches `image/segmentation`). `bbox` is `[x, y, w, h]` (COCO convention, matches `image/segmentation`). Keypoints below `threshold` are filtered out. Names come from the model's id2label when available; otherwise stringified indices.

### `POST /v1/images/detect`

Multipart input:

| field | type | required | notes |
|---|---|---|---|
| `image` | file | yes | as above |
| `model` | str | no | as above |
| `threshold` | float | no | Score threshold, default 0.5. |
| `max_detections` | int | no | Cap, default 100. |

Response:

```json
{
  "id": "det-<uuid>",
  "model": "detr-resnet-50",
  "image_size": [640, 480],
  "detections": [
    {"bbox": [120.0, 80.0, 200.0, 380.0], "score": 0.95, "label": "cat"}
  ]
}
```

Detections are sorted by score descending, capped at `max_detections`.

## Capability gating

Each model declares one of three flags in its manifest:

- `supports_depth: true` â†’ accepted by `/v1/images/depth`
- `supports_keypoints: true` â†’ accepted by `/v1/images/keypoints`
- `supports_detection: true` â†’ accepted by `/v1/images/detect`

Mismatch (e.g., depth model called via `/v1/images/keypoints`) returns 400 with code `wrong_primitive`. A model can declare multiple flags if it genuinely supports multiple primitives (none of our bundled or curated do; the flag is reserved for future multi-task models).

Additional advisory flag: `metric_depth: bool` (depth-only). Surfaced in `/v1/models` for client filtering.

## Runtimes

Three sibling runtimes in `src/muse/modalities/image_cv/runtimes/`:

1. **`HFDepthRuntime`** wraps `AutoModelForDepthEstimation` + `AutoImageProcessor`. Inference: `model(**inputs).predicted_depth` â†’ tensor of shape `(1, H, W)`. Post-processing: normalize to [0, 1] for relative depth (default), or pass through meters for metric depth.

2. **`HFKeypointRuntime`** wraps `AutoModelForKeypointDetection` + `AutoImageProcessor` (transformers 4.46+). For ViTPose specifically, the model expects a bounding box per pose to extract; v1 passes a single full-image bbox per call (the whole image is the "person to extract pose from"). Multi-person extraction needs a person detector first, deferred to a future v0.X+1.

3. **`HFObjectDetectionRuntime`** wraps `AutoModelForObjectDetection` + `AutoImageProcessor`. Standard post-processing via `image_processor.post_process_object_detection(outputs, target_sizes=[(H, W)], threshold=...)`.

All three:
- Use `muse.core.runtime_helpers` for `select_device`, `dtype_for_name`, `set_inference_mode`, `LoadTimer`.
- Lazy-import `torch`, the `AutoModelFor*` class, and `AutoImageProcessor` via module-level sentinels (the muse pattern; tests patch these directly).
- Take constructor kwargs `model_id`, `hf_repo`, `local_dir`, `device`, `dtype`, plus per-runtime defaults.

## HF plugin

One file `src/muse/modalities/image_cv/hf.py` with three-way dispatch in `_resolve` based on tags:

- `depth-estimation` â†’ `HFDepthRuntime`, `supports_depth=True`
- `keypoint-detection` â†’ `HFKeypointRuntime`, `supports_keypoints=True`
- `object-detection` â†’ `HFObjectDetectionRuntime`, `supports_detection=True`

Repo-name fallback for tag-less repos: `depth`, `dpt`, `zoedepth`, `pose`, `keypoint`, `vitpose`, `rtmpose`, `detr`, `yolos`, `rtdetr`. Matches per-primitive.

The `metric_depth` flag is auto-set when the resolved depth model's name contains `zoedepth` or `metric`; users can override in curated entries.

Priority **110**: tag-based, same slot as `image/ocr`, `image/segmentation`. Loses to file-pattern plugins (100); wins over `text/classification` catch-all (200).

## Out of scope

- **Person detection + pose** end-to-end pipeline. ViTPose v1 receives a single full-image bbox; users who want multi-person pose run object detection first, then keypoint per detection.
- **Instance segmentation** (Mask R-CNN family). Belongs to `image/segmentation` (which already exists).
- **Panoptic segmentation**. Same; if added, lands in `image/segmentation`.
- **3D pose / mesh extraction**. That's #108 (3D generation) territory.
- **Streaming / video**. One image per request.

## Test strategy

**Protocol + codec**: dataclass tests for `DepthResult`, `KeypointResult`, `ObjectDetectionResult`. Codec tests for PNG16 encode/decode round-trip, float32 base64 round-trip, keypoint structure, detection sort/cap.

**Runtimes (3)**: mocked AutoModel + AutoImageProcessor. Test the inference path's output shape, post-processing, threshold filtering (keypoint/detection), float32 vs relative depth handling.

**Routes (3)**: FakeModel + multipart upload. Capability gate 400s, threshold parameter handling, max_detections cap, OpenAI error envelope on backend failure.

**HF plugin**: tag dispatch (3 branches), repo-name fallback (multiple substrings per primitive), metric depth auto-detection, sniff exclusion of unrelated tags.

**Bundled scripts (3)**: manifest required keys, capability flags (one true per script), runtime inheritance.

**Smoke matrix**: 3 new bundled models join `.github/workflows/fresh-venv-smoke.yml`. Total runner additions ~600MB.
