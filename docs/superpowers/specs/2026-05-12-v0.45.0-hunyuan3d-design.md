> **REVISED IN IMPLEMENTATION (v0.45.0).** The original spec assumed Hunyuan3D-2 exposed a unified dual-direction pipeline. Step B1 API verification (the lesson from v0.43.0 / v0.44.0) discovered that text-to-3D is actually a *two-stage internal pipeline*: text generates an image via `HunyuanDiTPipeline`, then the image is lifted to a mesh via `Hunyuan3DDiTFlowMatchingPipeline`. The runtime structures this with two sentinels (shape eager-loaded, T2I lazy-loaded). For current runtime see `src/muse/modalities/model_3d_generation/runtimes/hunyuan3d.py` and the v0.45.0 + v0.45.1 release notes.

# v0.45.0 Hunyuan3D-2 Dual-Direction: Design

**Status:** Approved (pending user review of this spec)
**Target version:** v0.45.0
**Task:** #108 (3D generation runtime classes), Phase 3 of 4 (final implementation phase)
**Date:** 2026-05-12

## Goal

Add a `Hunyuan3DRuntime` to the existing `3d/generation` modality so the curated `hunyuan3d-2` entry actually works end-to-end. The runtime wraps Tencent's Hunyuan3D-2 SDK loaded via the standard family-registry `trust_remote_code=True` path. Hunyuan3D-2 is muse's first 3D runtime that supports BOTH `image_to_3d` and `text_to_3d` from a single model. After this release, task #108 is closed (Wonder3D remains deferred indefinitely per the v0.44.0 decision).

## Context

v0.41.0 shipped four curated 3D entries (TripoSR, Wonder3D, TRELLIS, Hunyuan3D-2). v0.43.0 through v0.44.0 added Shap-E and TRELLIS runtimes through the family-registry pattern. This release plugs the final runtime (Hunyuan3D-2) into the registry, completing the 3D modality's coverage of all curated entries except Wonder3D.

The pattern is now mechanical: one `_Family` entry, one runtime file, one implementation-time API verification step. The dispatcher does not change; only one new branch is added to it.

## Architecture

### Hunyuan3DRuntime

```python
# src/muse/modalities/model_3d_generation/runtimes/hunyuan3d.py

class Hunyuan3DRuntime:
    """Image-and-text-to-3D via Tencent's Hunyuan3D-2 SDK.

    First muse 3D runtime to support both directions from one model.
    Wraps the Tencent SDK loaded via trust_remote_code. Returns GLB
    bytes through trimesh export for both directions.
    """

    model_id: str
    supports_image_to_3d: bool = True
    supports_text_to_3d: bool = True

    def __init__(
        self,
        *,
        model_id: str,
        hf_repo: str,
        local_dir: str | None = None,
        device: str = "auto",
        dtype: str = "fp16",
        trust_remote_code: bool = True,
        seed: int | None = None,
        num_inference_steps: int = 30,
        guidance_scale: float = 5.0,
        **_: Any,
    ) -> None: ...

    def image_to_3d(
        self, image: Any, **kwargs: Any,
    ) -> list[Generation3DResult]: ...

    def text_to_3d(
        self, prompt: str, **kwargs: Any,
    ) -> list[Generation3DResult]: ...
```

Both `image_to_3d` and `text_to_3d` are implemented (no `NotImplementedError` raise for either). The two methods may share most of the pipeline setup and differ only in the conditioning step; implementation factors that out into a private helper if natural.

### API discovery (Step B1, same lesson as v0.44.0)

The exact Tencent Hunyuan3D-2 SDK API is not known at spec time. Likely shape based on the published Tencent repo structure:

```python
# Approximate; verified at implementation time against the real SDK.
from hy3dgen.shapegen import Hunyuan3DDiTFlowMatchingPipeline
pipe = Hunyuan3DDiTFlowMatchingPipeline.from_pretrained(
    "tencent/Hunyuan3D-2",
    # possibly: trust_remote_code=True
)
# image-to-3D:
mesh = pipe(image=pil_image, ...)
# text-to-3D:
mesh = pipe(prompt="...", ...)
```

The implementer's Step B1 verifies the real loader class name, the real pipeline call signature (one pipeline with both image and text, or two pipelines?), the output type, and the pip dependencies BEFORE writing mocks. Same discipline as v0.44.0 TRELLIS, which caught four plan-vs-reality mismatches.

_HUNYUAN3D_RUNTIME_PATH = (

    name_hints=("hunyuan3d",),

## Capability schema

No additions. The existing `supports_image_to_3d / supports_text_to_3d / trust_remote_code / output_format` from v0.43.x are sufficient.

## Error handling

- Missing torch / diffusers / trimesh / hy3dgen: constructor raises RuntimeError with actionable `muse models refresh hunyuan3d-2` hint.
- `image_to_3d` and `text_to_3d` both implement happy paths; no `NotImplementedError` raises for either.
- Pipeline load failure (network, corrupted snapshot): exception propagates; supervisor's worker spawns a fresh process on next request.
- Generation failure (CUDA OOM, malformed input, conditioning failure): exception propagates to the route layer (500 response).

## Risk register

Same shape as v0.44.0:

- **Pipeline class name uncertainty.** The exact loader (`Hunyuan3DDiTFlowMatchingPipeline`, `Hunyuan3D-2Pipeline`, or another) is not confirmed at spec time. The implementer verifies against the downloaded SDK in Step B1.
- **Single pipeline vs two.** Hunyuan3D-2 may expose one unified pipeline that handles both image and text inputs, OR two separate pipeline classes. The implementer adapts the runtime to whichever pattern the SDK uses.
- **pip dependencies.** The Hunyuan3D-2 SDK likely has native-build deps (kaolin, xformers, possibly custom CUDA extensions). The pip_extras include a git URL but Tencent's setup script may be required. Same caveat as TRELLIS.
- **GLB export shape.** Whether the SDK returns a `trimesh.Trimesh` directly, raw vertex/face arrays, or its own mesh type is verified at implementation time.

These risks are the third instance of the same pattern. The lesson from v0.43.0 → v0.43.1 and v0.44.0's B1 catches: verify the real API at implementation time, not the speculative one.
