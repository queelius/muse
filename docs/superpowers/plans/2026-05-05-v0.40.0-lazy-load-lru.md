# v0.40.0 Implementation Plan: Lazy Load + LRU Eviction

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace static eager-load with lazy first-request loading + on-demand LRU eviction backed by live `pynvml` / `psutil` measurements. Catalog capacity becomes effectively unlimited; the loaded set is sized to fit live free memory.

**Spec:** `docs/superpowers/specs/2026-05-05-v0.40.0-lazy-load-lru-design.md`.

**Tech stack:** existing supervisor + gateway + admin operations (reuse `enable_model` / `disable_model`); new `pynvml` (`nvidia-ml-py`) soft dep + existing `psutil` for memory probing.

### Task C: `LoadDirector` eviction loop

- [ ] **C1.** Write failing tests:
  - Tight budget: cold acquire with 1 evictable LRU candidate → director picks it, calls disable_fn, retries fit check, succeeds.
  - All loaded models have refcount > 0: cold acquire raises `OperationError("model_too_large_for_device", status=503)`.
  - Multi-eviction: shortfall larger than any single candidate → evicts in LRU order until fits.
  - Memory release polling: simulate `gpu_free_gb` returning rising values over poll iterations; eviction loop should converge.
- [ ] **C2.** Implement `_evict_lru_until_fits`. Helper signature: takes `shortfall_gb`, returns `True` if fit achieved, `False` (with structured reason) if not.
- [ ] **C3.** Implement `_wait_for_memory_release(min_freed_gb, timeout=2.0, poll=0.05)`: re-poll `memory_probe.gpu_free_gb()` (or cpu_free_gb) until either `freed >= min_freed_gb` OR timeout. Return actual freed amount.
- [ ] **C4.** Refcount safety: skip candidates where `refcount > 0`. Always log structured event for each eviction with the reason.
- [ ] **C5.** Run tests; commit: `feat(load_director): on-demand LRU eviction with memory-release polling`.

## Risks & rollback

**Concerns identified during spec/plan review:**

- Cold-start latency change is user-visible; mitigated by `muse models warmup <id>` and clear documentation.
- pynvml driver mismatches on operator boxes; mitigated by graceful fallback to declared budget cap.
- Worker SIGTERM-to-VRAM-release timing is driver-dependent; mitigated by 2s polling timeout + clean fallback to "spawn anyway, let OOM happen, surface clearly."

**Rollback strategy:** revert v0.39.x → v0.40.0 commits via `git revert` of the merge; catalog format is forward-compatible (the `unservable_reason` and `last_loaded_at` fields are unread by older muse).
